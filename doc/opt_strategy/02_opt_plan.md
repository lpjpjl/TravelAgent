## 二、优化方向汇总

优先级仅表示目标重要程度，不表示开发先后或实现难度：

- **P0**：最终目标不可缺少，缺失则核心产品不成立；
- **P1**：可靠性、安全性与规划质量的重要保障；
- **P2**：用户体验和工程效率优化；
- **P3**：低优先级工程治理；
- **Future**：可选增强，不纳入当前目标承诺。

具体开发顺序、阶段拆分和阶段验收由 `03_implement_plan.md` 定义。

### P0 — 核心能力提升（9 项）

> P0 表示目标优先级，表内顺序仅用于表达逻辑关系，不代表实施阶段。

| 编号 | 优化项 | 问题 | 说明 |
|------|--------|------|------|
| P0-1 | **AmapService 结构化解析** | 4 | 解析 MCPTool 文本为 Pydantic 对象，由 TripCoordinator 统一调用 AmapService，并将结构化候选数据传给 Agent |
| P0-2 | **标准化数据调用链** | 2 | 删除 Agent Prompt 中的 `[TOOL_CALL:...]` 文本格式；由 TripCoordinator 直接调用 AmapService 获取结构化数据，Agent 不直接负责高德工具调用 |
| P0-3 | **多偏好完整检索** | 1 | 用户最多选择 3 个偏好；TripCoordinator 必须同步串行覆盖全部已选标签，通过 AmapService 获取对应候选数据，再由 DestinationResearchAgent 统一筛选。当偏好包含"美食"时，增加美食目的地候选（与景点等权），其余偏好正常处理 |
| P0-4 | **结构化草案输出** | 5 | DestinationResearchAgent 的候选筛选结果和 ItineraryPlannerAgent 的 `ItineraryDraft` 必须通过 Pydantic 校验。HelloAgents 不视为原生返回模型：统一使用 JSON mode/`response_format` 适配器获取 JSON 字符串，再以 `extra="forbid"` 的 Pydantic 模型校验；失败仅允许有限重试。Agent 不直接生成最终 `TripPlan`，最终结果由 TripCoordinator 通过确定性代码组装并校验 |
| P0-5 | **餐饮候选检索与筛选** | 22 | AmapService 统一检索餐饮候选，DestinationResearchAgent 负责筛选：美食偏好开启时，输出美食级（景点级）+便餐级；美食偏好关闭时，仅输出便餐级。早餐默认在酒店餐厅解决并于 09:00 前完成；只有被明确选为美食级早餐时才安排外出就餐。餐饮安排交给 ItineraryPlannerAgent 参与草案决策与时间分配 |
| P0-6 | **RouteService（基础版）** | 22 | RouteService 是项目内部的路线业务适配层，通过 AmapService 同步串行调用高德路线规划 API，统一输出相邻点位的交通方式、距离、耗时和路径坐标。固定交通偏好只查询对应方式；选择“混合交通”时，RouteService 获取高德实际支持的多种路线候选，ItineraryPlannerAgent 只能根据真实候选的距离、耗时、换乘等信息选择候选 ID，不得生成路线事实 |
| P0-7 | **行程深度关联化** | 22 | TripCoordinator 按三级优先级闭环规划：第一优先级为景点和美食级餐点，二者独立检索并作为每日主要点位；第二优先级为独立检索的酒店，筛选时必须依附第一优先级点位的地理分布；第三优先级为便餐级餐点，必须依附景点、美食级餐点和酒店，并携带锚点与真实距离供 Planner 选择。ItineraryPlannerAgent 输出含点位优先级/删除顺序的草案，RouteService 提供真实路线，TimelineCalculator 生成时间轴。若超出时间窗口而删除点位，TripCoordinator 必须先重新校验空间锚点，必要时重新筛选受影响的酒店/便餐，再计算新相邻点位路线、时间轴和预算，并在有限次数内重新校验。每日在外时长、用餐和缓冲规则保持不变；天气列入 P1-5，行前须知列入 P2-9，首日/末日边界纳入 P0-8 |
| P0-8 | **可执行时间轴信息** | 22 | 输出 Day Timeline（离开酒店、到达/离开点位、路程耗时、独立缓冲、用餐、返回酒店、当日在外总时长）。普通早餐在酒店餐厅解决并于 09:00 前完成，09:00 从酒店直奔首个景点；安排美食级早餐时改为 08:00 从酒店出发，先到早餐点用餐再前往景点。非末日最晚 21:00 返回酒店；末日无论早餐类型均须在 18:00 前返回酒店并结束旅程。前端以**复合时间轴卡片**展示：左侧时间轴轨道 + 右侧点位卡片（保留实景图、名称、地址、游览/用餐时长、简介、评分、费用等现有信息），不允许退化为纯时间线 |
| P0-9 | **对话式行程迭代** | 21 | 删除现有手动编辑入口；用户只能基于当前会话中的现有行程，用自然语言提出白名单范围内的调整。涉及点位、酒店、路线、时间或预算的修改必须由后端重新计算并重新校验，不能只修改前端数据。当前结构化行程和对话修订不进入 PDF 历史；具体支持类型、分期与实现方案由 03 文档定义 |

### P1 — 可靠性、安全性与规划质量（6 项）

| 编号 | 优化项 | 问题 | 说明 |
|------|--------|------|------|
| P1-1 | **分级 Fallback 与结果状态** | 6 | 坐标只能通过高德地理编码或带来源缓存修复，禁止使用默认城市坐标、算法平移或 LLM 补写。计划结果使用 `complete / partial / failed` 表示完整性，并用独立的 `degraded` 标识记录 stale 缓存或降级路径，部分计划不得伪装成完整成功 |
| P1-2 | **分级错误处理** | 8 | 区分配置错误/网络超时/LLM 异常 |
| P1-3 | **请求缓存 + MCP 降级** | 4,9 | AmapService 优先使用有效缓存；实时调用失败时允许使用可追溯的 stale 缓存并显式标记降级。缓存介质、TTL 和清理策略由 03 文档根据技术验证结果确定；本阶段外部查询保持同步串行 |
| P1-4 | **地图信息增强** | 23 | 地图与时间轴信息强对应：分天路线颜色、酒店起终点、天序号标记、地图点位与时间轴联动高亮；前端路线数据必须来自 RouteService，底层由 AmapService 调用高德路线规划 API，禁止景点间直线连线，且可通过同起终点与策略在高德路线规划中复核 |
| P1-5 | **天气驱动决策** | 22 | AmapService 获取并提供结构化天气数据，由 ItineraryPlannerAgent 在数据源支持的预报范围内根据天气调整室内外点位选择；不得使用气候建议冒充天气预报，不新增独立 WeatherService |
| P1-6 | **地图 Key 风险治理** | 12 | 对高德地图 Web JS API Key 配置来源白名单、配额和最小权限，降低公开 Key 被滥用的风险 |

## 三、关键决策记录

| 决策项 | 结论 | 要点 |
|--------|------|------|
| 输入范围 | 最多 3 天、最多 3 个偏好 | `travel_days` 必须等于起止日期的自然日数量；交通方式、住宿类型和偏好均使用受控枚举，前后端共同校验 |
| 执行模型 | 同步串行 | 景点、天气、酒店、餐饮和路线等外部查询均按依赖顺序同步串行执行，本阶段不实现查询并发，优先保证正确性和可诊断性 |
| Agent 编排模式 | TripCoordinator + 2 核心 Agent | TripCoordinator 是普通编排代码，不作为 Agent。DestinationResearchAgent 合并景点、酒店、餐饮候选筛选职责；ItineraryPlannerAgent 负责输出包含日期、点位顺序、点位用途、建议时长和规划说明的 `ItineraryDraft`。天气、路线、时间、预算和校验分别下沉到确定性服务 |
| 最终数据责任 | TripCoordinator 组装 TripPlan | Agent 只输出结构化候选结果和 `ItineraryDraft`；TripCoordinator 根据来源 ID 回填事实数据，结合 RouteService、TimelineCalculator、BudgetCalculator 和 TimelineValidator 的结果，组装并返回唯一的最终 `TripPlan` |
| RouteService 定位 | 项目内部业务适配层 | 调用链为 `TripCoordinator → RouteService → AmapService → 高德 MCP → 高德路线规划 API`。固定交通偏好只查询指定方式；混合交通同步获取真实路线候选后，由 ItineraryPlannerAgent 以结构化输出选择候选 ID；RouteService 不生成或估算路线数据 |
| 规划优先级 | 一级主要点位 → 二级酒店 → 三级便餐 | 景点和美食级餐点为第一优先级；酒店为第二优先级并依附主要点位；便餐为第三优先级并依附主要点位和酒店。三类候选可独立检索，但必须按该优先级完成筛选和规划关联 |
| 超窗修复 | 删点后重新关联与计算 | `ItineraryDraft` 必须提供点位优先级/删除顺序；删除点位后先重新校验酒店/便餐锚点并按需重选，再查询新相邻点位路线并重算时间轴、预算和校验，禁止沿用失效依附关系或路段 |
| 数据真实性边界 | 事实不可生成、决策允许推理、缺失显式降级 | 外部事实必须保留来源；Agent 只能筛选已有候选并生成建议性内容；缺失字段使用 `null` 并标记 `unavailable`，不得为补全页面而编造 |
| 计划结果状态 | 完整性与降级状态分离 | `plan_status` 使用 `complete / partial / failed`；`degraded` 独立表示是否使用 stale 缓存或其他降级路径。可执行的部分结果可以返回，但必须明确标记，不能作为 `complete` 展示 |
| 预算感知系统 | ❌ 不采用 | 保留费用明细展示，不做主动约束 |
| 时间与体力 | ✅ 简化 | 每日在外时长从离开酒店到返回酒店计算，包含全部交通、游览、在外用餐和缓冲。非末日普通早餐方案上限 12 小时，美食级早餐方案上限 13 小时；末日 18:00 结束旅程为最高约束。上限不要求填满，每段真实交通之外至少保留 30 分钟独立缓冲，不做复杂体力建模 |
| 时间轴视图 | ✅ 采用 | 优先上线时间轴卡片（可执行时间信息），Gantt 作为后续增强 |
| 多方案对比 | 🔄 暂缓 | 预留接口 |
| 天气角色 | ✅ 升级 | 从信息展示 → 决策驱动 |
| 行前准备 + 预约 | 🔀 合并 | 「行前须知」板块 |
| LLM 温度 | 搜索 temp=0 / 规划 temp=0.3 | |
| 前端状态管理 | Pinia 管理当前会话 | Pinia 负责当前结构化行程、加载状态、对话状态、修订结果和跨组件共享数据；删除手动编辑功能。sessionStorage 仅作为当前标签页的临时恢复副本，不承担历史归档职责 |
| 当前行程导航 | 有当前行程才显示入口 | 行程页始终可返回首页；首页仅在 Pinia/sessionStorage 中存在有效当前行程时显示“返回当前行程”入口，初次进入或当前行程不存在时不显示 |
| 历史行程 | 来源于“导出行程为 PDF” | 前端只生成一次 PDF Blob，先通过浏览器下载，再上传同一 Blob 到后端固定磁盘目录；后端每次查询历史时扫描目录中的全部 `.pdf` 文件，前端仅提供历史 PDF 下载入口，不提供结构化恢复或继续编辑能力。未执行导出的行程不进入历史 |
| 结构化行程持久化 | ❌ 不采用数据库持久化 | 当前 TripPlan 仅服务于当前会话展示和对话迭代；关闭会话后可丢弃。对话修订由客户端提交当前 TripPlan 和完整性令牌，后端重新计算并返回新版 TripPlan，不保存结构化版本历史 |
| 地图组件 | 封装 TripMap.vue | |
| 地图路线策略 | ✅ 采用真实道路规划 | 地图使用 RouteService 返回的真实道路路线绘制，底层数据来自高德路线规划 API，禁止点位间直线连线；并与时间轴段落双向联动。路线结果需可复核，不强制要求持久化存储 |

| 多天信息展示 | ✅ 互不干扰 | 默认全部折叠，支持多天同时展开；不再采用排他式手风琴 |

### 3.1 每日在外时长与缓冲约束

每日在外时长定义为：

```text
每日在外时长 = 返回酒店时间 - 离开酒店时间
```

统计范围包含：

- 酒店到首个点位的交通；
- 景点游览和在外餐饮活动；
- 点位之间的全部交通；
- 为突发情况预留的独立缓冲；
- 最后一个点位返回酒店的交通。

每日时间窗口由早餐类型和是否末日共同决定：

| 日期类型 | 早餐类型 | 离开酒店 | 最晚返回 | 在外时长上限 |
|----------|----------|----------|----------|--------------|
| 非末日 | 酒店早餐 | 09:00 | 21:00 | 12 小时 |
| 非末日 | 美食级早餐 | 08:00 | 21:00 | 13 小时 |
| 末日 | 酒店早餐 | 09:00 | 18:00 | 9 小时 |
| 末日 | 美食级早餐 | 08:00 | 18:00 | 10 小时 |

上述时长均为硬上限，不是必须填满的规划额度。行程仍以每天 2～3 个主要点位为宜；候选不足或继续安排会降低行程质量时，允许提前返回酒店，禁止为了填满时长增加低价值点位。

点位间过渡时段定义为：

```text
点位间过渡时段 = RouteService 返回的真实交通耗时 + 独立缓冲时间
独立缓冲时间 ≥ 30 分钟
```

独立缓冲用于应对堵车、排队、找路、临时休息等突发情况，必须在时间轴中以 `buffer` 段显式保留。缓冲不能预先用于购物、拍照、额外景点或其他计划内活动，也不能被交通耗时替代。

时间边界规则：

- 早餐不是美食级餐饮时，统一在酒店餐厅解决，并认为 09:00 时已经用餐完毕，离开酒店后直奔首个景点；酒店早餐不计入在外时长；
- 早餐被明确安排为美食级餐饮时，08:00 从酒店出发，先前往早餐点用餐，再前往首个景点；酒店到早餐点、早餐用餐、早餐点到景点的交通和缓冲全部计入在外时长；
- 首日按完整旅行日处理：普通早餐方案默认 09:00 前已入住酒店，美食级早餐方案默认 08:00 前已入住酒店；
- 非末日最晚 21:00 返回酒店；末日 18:00 返回酒店并结束旅程是最高约束，不因美食级早餐而顺延；
- 单日行程同时属于首日和末日，适用末日 18:00 结束规则；
- 所有日期均按三餐规划，末日晚餐必须在 18:00 旅程结束前完成；
- 无法在窗口内完成时，减少低优先级点位，不得压缩真实交通、最低用餐时长或独立缓冲。

## 四、数据真实性边界与来源规则

### 4.1 基本原则

1. **事实数据不可生成**：POI、地址、坐标、路线、天气等事实必须来自明确的数据源，LLM 不得补写。
2. **决策数据允许推理**：候选筛选、偏好匹配、分天排序等允许 Agent 判断，但只能操作已有候选 ID。
3. **计算结果由程序负责**：时间轴时刻、预算合计等由确定性代码计算，Agent 不直接输出最终 `TripPlan`。
4. **缺失数据显式降级**：字段缺失时使用 `null` 并标记 `unavailable`，不得使用虚构值维持页面完整。
5. **外部来源不等于绝对正确**：来自高德等外部服务的数据统一标记为 `sourced`，不使用可能造成过度承诺的 `verified`。

### 4.2 数据类型

| 类型 | 含义 | 示例 |
|------|------|------|
| `sourced` | 来自可追溯的外部服务或用户输入 | POI、坐标、路线、天气、用户偏好 |
| `computed` | 由程序按确定性规则计算 | 日期、预算汇总、到达/离开时刻 |
| `estimated` | Agent 或确定性规则给出的估算值 | Agent 建议的游览时长、BudgetCalculator 按规则生成的预估费用 |
| `generated` | LLM 生成的建议性内容 | 景点简介、旅行建议、选择理由、行前须知 |
| `unavailable` | 当前数据源没有提供 | 缺失的票价、评分、营业时间或远期天气 |

### 4.3 字段责任边界

| 数据 | 责任方 | 约束 |
|------|--------|------|
| POI ID、名称、地址、坐标、类型 | AmapService | 必须来自高德返回结果，Agent 不得新增不存在的 POI |
| 景点、酒店、餐厅候选 | AmapService | 负责检索并保留来源 ID；无结果时返回空集合 |
| 景点、酒店、餐厅筛选 | DestinationResearchAgent | 只能返回候选集合中已存在的 ID |
| 路线、距离、交通耗时、路径坐标 | RouteService | 底层来自高德路线规划 API，禁止 LLM 估算或补写 |
| 天气 | AmapService | 获取并结构化高德天气数据；仅展示数据源支持日期范围内的预报，超出范围标记 `unavailable`，气候建议不得冒充天气预报 |
| 评分、营业时间、票价 | 外部服务 | 数据源未返回时保持 `null`，不得由 LLM 补全 |
| 分天安排、点位排序、点位用途、点位优先级/删除顺序、建议游览时长 | ItineraryPlannerAgent | 输出 `ItineraryDraft`；点位必须来自候选 ID，优先级必须遵循主要点位→酒店→便餐规则，游览时长标记为 `estimated` |
| 混合交通路线选择 | ItineraryPlannerAgent | RouteService 先提供同一路段的真实路线候选；Agent 只能返回候选 route ID 和选择理由，不得改写距离、耗时、换乘或路径坐标 |
| 到达/离开时刻、时间冲突检查 | TimelineCalculator / TimelineValidator | 根据真实路线耗时、游览时长和缓冲规则确定性计算与校验 |
| 单项预估费用 | BudgetCalculator | 外部事实价格缺失时，仅可按明确的确定性规则估算，必须标记 `estimated` 并保留估算依据；没有可用规则时标记 `unavailable` |
| 预算汇总 | BudgetCalculator | 根据有效的事实价格与规则估算项确定性计算；BudgetCalculator 是所有单项预估费用和预算汇总字段的唯一提供方 |
| 简介、总体建议、选择理由、行前须知 | ItineraryPlannerAgent | 允许生成，统一标记为 `generated`，不得夹带未经来源支持的事实 |

`ItineraryDraft` 只允许包含：日期、有序点位 ID、点位用途（景点/早餐/午餐/晚餐）、规划优先级、删除顺序、建议活动时长和规划说明。混合交通另使用受限的 `RouteSelection` 模型，仅允许选择 RouteService 已提供的 route ID。上述 Agent 输出均使用 JSON mode/`response_format` 适配器，并由 `extra="forbid"` 的 Pydantic 模型校验，失败时有限重试。它们不得包含或改写地址、坐标、真实路线、距离、交通耗时、最终到达/离开时刻、事实价格、单项预估费用或预算合计。

最终数据流固定为：

```text
AmapService（同步独立检索）
  → DestinationResearchAgent（一级主要点位 → 二级酒店 → 三级便餐）
  → ItineraryPlannerAgent(ItineraryDraft)
  → RouteService（固定方式；混合方式返回多个真实候选）
  → ItineraryPlannerAgent(RouteSelection，仅混合交通)
  → TimelineCalculator / BudgetCalculator → TimelineValidator
  → 不通过时删点并重算受影响路线（有限循环）
  → TripCoordinator → TripPlan
```

### 4.4 来源元数据

不对每个字段包装 `{value, source}`，避免前后端模型过度复杂。景点、酒店、餐厅、天气和路线等事实实体统一增加轻量来源元数据：

```json
{
  "name": "故宫博物院",
  "address": "北京市东城区景山前街4号",
  "ticket_price": null,
  "data_quality": {
    "source": "amap",
    "source_id": "B000A8UIN8",
    "fetched_at": "2026-06-29T10:00:00+08:00",
    "freshness": "live",
    "estimated_fields": ["visit_duration"],
    "generated_fields": ["description"],
    "unavailable_fields": ["ticket_price"]
  }
}
```

`freshness` 取值限定为：

- `live`：本次请求实时获取；
- `cached`：命中有效期内缓存；
- `stale`：实时调用失败后使用过期缓存，必须向用户提示；
- `not_applicable`：计算值或生成内容不适用新鲜度概念。

### 4.5 防止 Agent 编造的程序约束

1. AmapService 返回带稳定来源 ID 的结构化候选集合。
2. DestinationResearchAgent 只输出候选 ID 和筛选理由，不回写地址、坐标等事实字段。
3. TripCoordinator 根据候选 ID 回填事实实体；Agent 返回未知 ID 时拒绝结果并触发有限重试或降级。
4. ItineraryPlannerAgent 只能使用 TripCoordinator 提供的点位 ID 生成 `ItineraryDraft`，不得创建新点位或输出最终 `TripPlan`。
5. RouteService 只接收已有点位坐标，并且是路线、距离和交通耗时字段的唯一提供方。
6. TimelineCalculator 根据草案与事实数据确定性生成时间轴；BudgetCalculator 根据事实价格和明确规则生成单项预估费用与预算汇总。时间和费用字段均不接受 Agent 输出。
7. TimelineValidator 校验未知 POI、缺失路线、伪造坐标与时间冲突，不通过时不得作为正常计划返回。
8. 只有 TripCoordinator 可以组装并返回最终 `TripPlan`，前端不得消费 Agent 草案。

### 4.6 数据降级规则

```text
实时数据成功       → sourced / live
实时失败且缓存有效  → sourced / cached
仅有过期缓存       → sourced / stale，degraded=true，并展示降级提示
无真实候选或关键路线 → 返回仍可执行的 partial，无法形成可执行计划时返回 failed
```

坐标修复只能使用高德地理编码结果或带来源的有效缓存。Fallback 不再生成“某城市景点1”、北京默认坐标、算法平移坐标或其他虚构事实。

计划完整性状态定义为：

- `complete`：请求范围内的每日计划均可执行，所有已安排点位具有合法来源，所需路线和时间轴完整；票价、评分等可选事实缺失不影响完整性；
- `partial`：仅能返回部分日期或部分必要安排，但返回部分本身必须真实、可执行，并明确列出缺失项；
- `failed`：无法形成任何满足真实性和基本可执行约束的计划，不返回伪造兜底数据。

`degraded` 与 `plan_status` 相互独立：使用 stale 缓存或其他降级数据路径时设为 `true`。因此结果可以是 `complete + degraded` 或 `partial + degraded`，但不能将 `partial` 作为 `complete` 展示。

### 4.7 验收规则

1. 所有景点、酒店和餐厅 ID 均能在本次候选集合或带来源的缓存中找到。
2. 所有路线段均由 RouteService 返回，不存在 LLM 生成的距离、耗时或路径坐标。
3. 事实字段缺失时为 `null`，并出现在 `unavailable_fields` 中。
4. BudgetCalculator 输出的预算合计与有效明细计算结果一致。
5. 超出天气数据支持范围的日期不展示伪预报。
6. `stale` 数据、部分计划和其他降级结果均有明确状态与用户提示。
7. `complete / partial / failed` 与 `degraded` 的组合符合定义，前端不得将部分或失败结果展示为完整成功。

## 五、高德 API 依赖与验证约束

`RouteService` 不是高德提供的独立服务，而是项目内部封装。它复用 `AmapService` 接入高德路线规划能力，并向 TripCoordinator 和前端提供统一的业务数据结构。

单次行程的 API 调用量取决于偏好数量、候选数量、旅行天数、路线段数量与缓存命中情况，不能以固定次数作为设计前提。高德官方配额、计费规则和字段能力可能调整，也不能在 02 文档中承诺固定日配额、可支持规划次数或所有调用均免费。

03 文档必须安排技术验证，至少确认：

- 目标版本 `amap-mcp-server` 的实际工具列表与返回结构；
- POI 来源 ID、路线距离、耗时和路径坐标是否稳定可用；
- 同步串行调用下 MCP 实例的连续调用稳定性与资源释放行为；
- 目标高德 Key 的实时官方配额、限流和计费规则；
- 典型 1～3 日行程的实际调用量与失败率。

缓存、限流和降级方案必须以验证结果为依据，不得依赖未经确认的固定数字。本阶段不实施查询并发；除配额外，还需关注 `amap-mcp-server` 第三方中间层的兼容性与稳定性。

**对策：AmapService 缓存降级**

```
search_poi("故宫", "北京")
  ├─ 查有效缓存 → 命中 ✅
  ├─ 未命中 → MCP → 高德 → 解析 + 写缓存
  └─ MCP 异常 → stale 缓存兜底 ⚠️ / 触发分级降级
```

缓存介质、各类数据 TTL、容量清理和持久化路径均不在 02 文档中固定，由 03 文档根据技术验证结果与阶段目标确定。

### 6.3 设计验收标准

1. **不退化**：每个景点/美食目的地保留现有信息区域；评分、费用等事实缺失时显示“暂无数据”，不得为维持信息密度而生成虚构值。
2. **可执行**：用户能依次读出一条路径——几点出发、怎么去、多久到、玩多久/吃多久、中间有没有缓冲。
3. **可对比**：用户手动同时展开多天后，不同天的时间轴可纵向对齐对比。
4. **可联动**：hover 时间轴某段 → 地图对应路线高亮，反之亦然。
5. **两级餐饮**：美食偏好开启时，美食目的地 = 景点级卡片（同模板）；未安排美食级早餐时，早餐固定显示为酒店餐厅且不进入外出时间轴；所有实际在外用餐均在时间轴预留 ≥40min。
6. **可追溯**：RouteService 给出的每段路线都可用同起终点与策略在高德路线规划 API 中复核（可查得到），禁止使用 LLM 编造路线；不强制要求路线来源持久化。
7. **设备范围**：本阶段仅验收 PC 端显示与交互完整性，不纳入移动端/平板适配要求。
8. **在外时长**：非末日酒店早餐方案不超过 12 小时，美食级早餐方案不超过 13 小时；末日分别不超过 9 小时和 10 小时，且 18:00 结束旅程优先于其他规则。所有上限均不得视为必须填满的额度。
9. **独立缓冲**：每段真实交通之外至少保留 30 分钟 `buffer`，交通耗时不得替代缓冲，缓冲也不得预先安排其他活动。
10. **早餐规则**：酒店早餐必须在 09:00 前完成并从酒店直奔景点；美食级早餐必须 08:00 出发，先完成早餐点的交通、缓冲和用餐，再前往景点。
