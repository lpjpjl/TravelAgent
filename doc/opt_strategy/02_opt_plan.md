## 二、优化方向汇总

优先级仅表示目标重要程度，不表示开发先后或实现难度：

- **P0**：最终目标不可缺少，缺失则核心产品不成立；
- **P1**：可靠性、安全性与规划质量的重要保障；
- **P2**：用户体验和工程效率优化；
- **P3**：低优先级工程治理；
- **Future**：可选增强，不纳入当前目标承诺。

具体开发顺序、阶段拆分和阶段验收由 `03_implement_plan.md` 定义。

### P0 — 核心能力提升（9 项）

> P0 表示目标优先级，表内顺序仅用于表达逻辑关系，不代表实施阶段。

| 编号 | 优化项 | 问题 | 说明 |
|------|--------|------|------|
| P0-1 | **AmapService 结构化解析** | 4 | AmapService 将高德数据统一解析为 Pydantic 对象，由 TripCoordinator 调用并将结构化候选传给 Agent。默认先验证并使用 MCP；若目标 MCP 在字段完整性、稳定性或资源释放方面不能满足要求，允许 AmapService 直接调用高德官方 HTTP API。两种接入方式必须输出同一领域模型并遵守相同真实性、错误和缓存语义，上层不得感知数据通道差异 |
| P0-2 | **标准化数据调用链** | 2 | 删除 Agent Prompt 中的 `[TOOL_CALL:...]` 文本格式；由 TripCoordinator 直接调用 AmapService 获取结构化数据，Agent 不直接负责高德工具调用 |
| P0-3 | **多偏好完整检索** | 1 | 用户最多选择 3 个偏好；TripCoordinator 必须同步串行覆盖全部已选标签，通过 AmapService 获取对应候选数据，再由 DestinationResearchAgent 统一筛选。当偏好包含"美食"时，增加美食目的地候选（与景点等权），其余偏好正常处理。首页“额外要求”不是普通偏好：经结构化判定后作为最高用户约束优先级，可触发必打卡 POI 精确检索或餐饮禁忌过滤 |
| P0-4 | **结构化草案输出** | 5 | DestinationResearchAgent 先将首页“额外要求”解析为封闭枚举的 `ExtraConstraint`；明确子句进入规划，无法定性、语义模糊或当前不支持的子句直接记录为 ignored 并丢弃。候选筛选结果、约束解析结果、ItineraryPlannerAgent 的主要点位 `PrimaryItineraryDraft` 和完整 `ItineraryDraft` 均须通过 Pydantic 校验。HelloAgents 不视为原生返回模型：统一使用 JSON mode/`response_format` 适配器获取 JSON 字符串，再以 `extra="forbid"` 的 Pydantic 模型校验；失败仅允许有限重试。Agent 不直接生成最终 `TripPlan`，最终结果由 TripCoordinator 确定性组装并校验 |
| P0-5 | **餐饮候选检索与筛选** | 22 | AmapService 统一检索餐饮候选，DestinationResearchAgent 负责筛选：美食偏好开启时，完整计划为 1～N 个、partial 为 1～K 个美食级餐点且每天最多 1 个。美食级餐点按 `meal_role` 占用正式餐次，同餐次不得再安排便餐；缺失午晚餐使用高德真实便餐 POI。营业时间有来源值时，美食级/便餐级午晚餐在候选阶段校验固定餐窗内存在连续 ≥40min 可用区间；早餐没有固定候选餐窗，美食级早餐在路线和时间轴生成后按实际用餐区间校验。营业时间缺失时允许使用，但标记 `unavailable` 并提示核实 |
| P0-6 | **RouteService（基础版）** | 22 | RouteService 通过 AmapService 同步串行调用高德路线规划 API。交通方式是默认偏好，不是系统硬约束：先查询偏好方式，不可用或不合格时自动查询其他无需额外前提的真实方式；公共交通不回退为自驾，只有用户选择自驾时才使用 `self_drive`。步行只适用于真实路线距离 ≤2km 的路段，超限时强制回退打车/公共交通。回退候选按真实耗时最短确定性选择，耗时相同再按距离和 route ID 稳定决胜。混合交通候选包含合格步行、打车、地铁/公交和数据源明确支持的旅游专线大巴，不包含自驾。accepted 移动约束仍是硬约束 |
| P0-7 | **行程深度关联化** | 22 | TripCoordinator 采用两次草案闭环。`PrimaryItineraryDraft` 中，景点等非餐饮主要点标记 `day_part=morning|afternoon`，美食级餐点标记`meal_role=breakfast|lunch|dinner` 并进入对应正式用餐窗口；provisional meal slots 记录已由美食级餐点占用或仍需便餐补齐的餐次。连续住宿和缺失餐次的真实便餐选定后再生成完整 `ItineraryDraft`。TimelineCalculator 只报告问题；TripCoordinator 按问题类型依次尝试替换路线、替换餐厅、因明确时间约束重排 PrimaryDraft，最后才删除可选主要点并重建全部下游。酒店、三餐和必打卡点不可删除；任何 accepted 约束无法满足时整单 failed |
| P0-8 | **可执行时间轴信息** | 22 | 输出 Day Timeline。午餐必须在 12:00—14:00 内完成，普通日晚餐必须在 18:00—20:00 内完成，末日晚餐调整为 17:00—18:00；每顿仍不少于 40 分钟。午餐实际开始前结束的非餐饮主要点属于上午，午餐实际结束后开始的非餐饮主要点属于下午，点位不得跨越午餐；美食级餐点直接进入其 `meal_role` 对应窗口。非末日最晚 21:00 抵达 `end_hotel_id`；末日晚餐结束即旅程结束。前端以复合时间轴卡片展示起止酒店、活动、真实路线和缓冲 |
| P0-9 | **对话式行程迭代** | 21 | 删除手动编辑；首版支持独立 intent：更换美食级餐点、更换便餐、修改美食级 `meal_role`，以及既定的点位、酒店、日期/顺序和交通偏好调整。当前行程的 normalized accepted 约束在生命周期内不可修改并由完整性令牌保护；修订与其冲突时不执行、不写入已采纳对话，前端保留输入供修改，在文字下显示“与额外要求冲突，不予采纳”且输入框整体红边，用户一修改即清除提示并恢复边框。实质修改必须由后端重算校验；修改 accepted 约束须回首页重新生成 |

### P1 — 可靠性、安全性与规划质量（6 项）

| 编号 | 优化项 | 问题 | 说明 |
|------|--------|------|------|
| P1-1 | **分级 Fallback 与结果状态** | 6 | 坐标只能通过高德地理编码或带来源缓存修复，禁止使用默认城市坐标、算法平移或 LLM 补写。计划结果使用 `complete / partial / failed` 表示完整性，并用独立的 `degraded` 标识记录 stale 缓存或降级路径，部分计划不得伪装成完整成功 |
| P1-2 | **分级错误处理** | 8 | 区分配置错误、网络超时、上游高德/MCP、核心 Agent/草案解析和业务不可行：有效响应但没有可行候选属于业务不可行，可进入 partial/failed；核心规划链路重试及缓存降级均失败、无法形成可信规划输入时返回分级 API 错误。额外要求解析失败是明确例外，按 `parse_failed` 进入 ignored 后继续规划 |
| P1-3 | **请求缓存 + 外部数据降级** | 4,9 | AmapService 优先使用有效缓存；无论实时通道是 MCP 还是高德官方 HTTP API，调用失败时均允许使用可追溯的 stale 缓存并显式标记降级。当前单进程单实例范围允许使用进程内缓存；TTL、容量和清理策略由 03 文档根据技术验证结果确定。本阶段外部查询保持同步串行 |
| P1-4 | **地图信息增强** | 23 | 地图与时间轴信息强对应：分天路线颜色、酒店起终点、天序号标记、地图点位与时间轴联动高亮；前端路线数据必须来自 RouteService，底层由 AmapService 调用高德路线规划 API，禁止景点间直线连线，且可通过同起终点与策略在高德路线规划中复核 |
| P1-5 | **天气驱动决策** | 22 | AmapService 获取结构化天气；在预报范围内，DestinationResearchAgent 先基于带室内外标签的真实候选重新筛选/替换主要点位，ItineraryPlannerAgent 再负责分天与排序。天气导致点位集合变化时必须重建 PrimaryDraft 及全部下游；不得用气候建议冒充天气预报，不新增 WeatherService |
| P1-6 | **地图 Key 风险治理** | 12 | 当前仅支持本地开发来源，对高德地图 Web JS API Key 配置 localhost/127.0.0.1 来源限制、配额和最小权限，降低公开 Key 被滥用的风险；未来增加正式域名时必须显式更新来源白名单，不预留任意来源通配符 |

## 三、关键决策记录

| 决策项 | 结论 | 要点 |
|--------|------|------|
| 当前环境范围 | 本地开发环境 | 本轮复用 `.env` 中已经配置的 LLM，不设计生产模型、生产密钥或多环境切换；不得把 `.env` 中的真实密钥写入文档、日志、前端构建产物或测试 fixture |
| 运行拓扑 | 单机、单进程、单实例 | 缓存、SSE、限流和本地文件访问按单进程语义设计，不引入 Redis、消息队列、分布式锁或共享任务状态；拓扑变化时必须重新评审这些能力 |
| 高德接入通道 | MCP 优先验证，官方 HTTP API 可回退 | AmapService 是唯一高德数据边界。若 MCP 无法稳定提供所需字段或连续调用不可靠，允许直接调用高德官方 HTTP API；通道切换不得改变上层领域模型、来源标记、错误码和缓存键语义 |
| 地图访问来源 | 仅本地开发来源 | Web JS API Key 只允许实际使用的 localhost/127.0.0.1 开发来源，不配置任意来源通配符；后端高德 Key 不进入前端 |
| PDF 存储前提 | 固定持久化磁盘 | PDF 目录在服务重启后保持不变，应用不得将容器临时目录或系统临时目录作为正式报告目录；写入、扫描和下载均限定在配置目录内 |
| 输入范围 | 最多 3 天、最多 3 个偏好 | `travel_days` 必须等于起止日期的自然日数量；交通方式、住宿类型和偏好均使用受控枚举，前后端共同校验 |
| 额外要求优先级 | 最高用户约束优先级 | 首页 `free_text_input` 中能够明确归类的需求优先于旅行偏好、三级选址优先级和普通 Planner 建议。例如“不吃辣”先过滤不符合条件的餐饮候选，“龙门石窟是必打卡点”先精确检索并将真实匹配 POI 标记为 required。数据真实性、时间窗口、必要路线等系统硬约束仍不可被额外要求覆盖 |
| 模糊额外要求 | 测试期直接丢弃 | 当前测试阶段由开发者保证输入为确定需求；DestinationResearchAgent 若仍无法将某个需求子句明确归类并结构化，则将该子句记入 `ignored_extra_requirements` 后直接丢弃，不追问、不猜测、不对候选或规划施加约束。可明确解析的其他子句继续生效 |
| 执行模型 | 同步串行 | 景点、天气、酒店、餐饮和路线等外部查询均按依赖顺序同步串行执行，本阶段不实现查询并发，优先保证正确性和可诊断性 |
| Agent 编排模式 | TripCoordinator + 2 核心 Agent | TripCoordinator 是普通编排代码。DestinationResearchAgent 分别负责额外要求解析、主要点位筛选、酒店与便餐筛选；ItineraryPlannerAgent 第一次输出 `PrimaryItineraryDraft`，第二次输出完整 `ItineraryDraft`，混合交通时另输出 `RouteSelection`。天气、路线、时间、预算和校验下沉到确定性服务 |
| 最终数据责任 | TripCoordinator 组装 TripPlan | Agent 只输出结构化约束、候选 ID、`PrimaryItineraryDraft`、`ItineraryDraft` 和受限路线选择；TripCoordinator 根据来源 ID 回填事实数据，结合 RouteService、TimelineCalculator、BudgetCalculator 和 TimelineValidator，组装并返回唯一 `TripPlan` |
| RouteService 定位 | 项目内部业务适配层 | 所有交通选项均为默认偏好：偏好方式不可用或被硬约束过滤后，自动查询其他无需额外前提的真实方式并按耗时、距离、route ID 确定性选择；公共交通偏好不假设用户有车，因此不回退到自驾。步行路线 ≤2km 才可选，>2km 强制回退打车/公共交通；公共交通的实际步行接驳段也分别满足上限。只有 accepted 移动约束不可放宽；混合交通不包含自驾 |
| 规划优先级 | PrimaryDraft → 连续酒店 → 缺失餐次便餐 → 完整草案 | 景点和美食级餐点先完成筛选、分天与排序；美食级餐点按 `meal_role` 直接占用对应餐次。再选择 N 个连续住宿记录，最后只为尚未被美食级餐点占用的午餐/晚餐检索便餐。便餐选址优先级最低但一旦用于补齐餐次即为 required，必须进入完整草案、路线、时间轴和预算 |
| 酒店使用规则 | N 个连续住宿记录，计费 N 晚 | N 天对应 `lodging[0..N-1]`：`lodging[0]` 是首日前一晚酒店；第 i 天从 `lodging[i]` 出发，非末日抵达 `lodging[i+1]`，末日无结束酒店。酒店可相同或不同；A→B 表示当日结束时换到 B 入住，下一日必须从 B 出发。本阶段不单列退房、行李搬运和入住耗时，但必须计算最后活动到结束酒店的真实路线 |
| 修复顺序 | Coordinator 按问题类型修复 | TimelineCalculator/Validator 不修改草案。步行超限或移动约束冲突先替换真实路线；餐饮锚点/时间冲突再按真实路线代价替换同级餐厅；酒店位置导致必要路线缺失或时间轴不可行时替换同级酒店，并重建相邻日期的酒店边界和下游；`explicit_time_constraint` 冲突回到 PrimaryDraft 调整日期、顺序、非餐饮点 `day_part` 或美食级餐点所在餐次位置；普通超窗最后才按 `removal_rank` 删除可选主要点。不得用坐标直线距离宣称“更近”；每次修复后重建受影响住宿、餐次、路线、时间轴和预算 |
| 数据真实性边界 | 事实不可生成、决策允许推理、缺失显式降级 | 外部事实必须保留来源；Agent 只能筛选已有候选并生成建议性内容；缺失字段使用 `null` 并标记 `unavailable`，不得为补全页面而编造 |
| 计划结果状态 | 完整性与降级状态分离 | Coordinator 必须先在完整日期范围内跨日重排主要点位、餐次和住宿并完成有限修复，仍无法形成完整计划时才允许生成 `partial`。partial 只能返回从首日开始、日期连续的最长可执行前缀 `Day1..DayK`；首日失败则整单 failed。若开启美食偏好，最终前缀必须含 1～K 个美食级餐点，否则先尝试把美食级餐点重排进前缀，仍失败则整单 failed。任一 accepted 约束无法满足时同样整单 failed |
| 预算口径 | 展示固定假设，不主动约束 | 默认 2 名成人、1 间房；住宿晚数统一取 `lodging_count=len(lodging)`：完整 N 天为 N 晚，partial Day1..DayK 且 K<N 时为 K+1 晚。门票、餐饮、公共交通按 2 人计算，打车按 1 车计算；页面展示实际 `lodging_count` 和交通计价单位 |

### 3.1 每日行程时长与缓冲约束

每日行程时长按是否末日分别定义：

```text
非末日行程时长 = 抵达 end_hotel 时间 - 离开 start_hotel 时间
末日行程时长   = 晚餐结束时间 - 离开 start_hotel 时间
```

统计范围包含：

- `start_hotel_id` 到首个点位的交通；
- 景点游览和在外餐饮活动；
- 点位之间的全部交通；
- 为突发情况预留的独立缓冲；
- 非末日最后一个点位到 `end_hotel_id` 的交通；末日不包含返店段。

每日时间窗口由早餐类型和是否末日共同决定：

| 日期类型 | 早餐类型 | 离开酒店 | 最晚结束 | 行程时长上限 |
|----------|----------|----------|----------|--------------|
| 非末日 | 住宿侧早餐 | 09:00 | 21:00 抵达 end_hotel | 12 小时 |
| 非末日 | 美食级早餐 | 08:00 | 21:00 抵达 end_hotel | 13 小时 |
| 末日 | 住宿侧早餐 | 09:00 | 18:00 晚餐结束 | 9 小时 |
| 末日 | 美食级早餐 | 08:00 | 18:00 晚餐结束 | 10 小时 |

上述时长均为硬上限，不是必须填满的规划额度。行程仍以每天 2～3 个主要点位为宜，但每个完整可执行日必须至少包含 1 个主要点位；景点或美食级餐点均计为主要点位，便餐不计。候选不足或继续安排会降低质量时，可只保留 1 个主要点位；非末日允许提前抵达结束酒店，末日允许在完成晚餐后提前结束，禁止为了填满时长增加低价值点位。

点位间过渡时段定义为：

```text
点位间过渡时段 = RouteService 返回的真实交通耗时 + 独立缓冲时间
独立缓冲时间 ≥ 30 分钟
```

独立缓冲用于应对堵车、排队、找路、临时休息等突发情况，必须在时间轴中以 `buffer` 段显式保留。缓冲不能预先用于购物、拍照、额外景点或其他计划内活动，也不能被交通耗时替代。

时间边界规则：

- 早餐不是美食级餐饮时，按“住宿侧解决”处理，界面提示“酒店餐厅或附近早餐店”；是否含早及具体地点以酒店实际政策为准。规划上认为 09:00 前已经用餐完毕，离开酒店后直奔首个主要点位，该早餐不作为已核实外部 POI，也不计入行程时长；
- 早餐被明确安排为美食级餐饮时，08:00 从酒店出发，先前往早餐点用餐，再前往首个景点；酒店到早餐点、早餐用餐、早餐点到景点的交通和缓冲全部计入行程时长；
- 首日按完整旅行日处理：普通早餐方案默认 09:00 前已入住酒店，美食级早餐方案默认 08:00 前已入住酒店；
- 非末日最晚 21:00 抵达 `end_hotel_id`；该酒店自动成为下一日 `start_hotel_id`。末日不设置结束酒店，晚餐必须在 18:00 前结束，该边界不因美食级早餐而顺延；
- 单日行程同时属于首日和末日，适用末日 18:00 结束规则；
- 所有日期均按三餐规划，末日晚餐必须在 18:00 旅程结束前完成；
- 午餐活动的 `start_at >= 12:00` 且 `end_at <= 14:00`；普通日晚餐 `start_at >= 18:00` 且 `end_at <= 20:00`；末日晚餐 `start_at >= 17:00` 且 `end_at <= 18:00`；
- 每顿在外用餐至少 40 分钟。非餐饮主要点 `end_at <= lunch.start_at` 时属于上午，`start_at >= lunch.end_at` 时属于下午；任何非餐饮主要点都不得与午餐重叠或跨越午餐时段。`meal_role=lunch` 的美食级餐点就是午餐活动本身，不适用该非餐饮点校验；
- 无法在窗口内完成时，减少低优先级点位，不得压缩真实交通、最低用餐时长或独立缓冲。

### 3.2 酒店与预算计算约束

- N 天行程固定包含 N 个住宿记录并计算 N 晚、每天 1 间房：`lodging[0]` 对应首日前一晚，`lodging[N-1]` 对应末日前一晚，末日晚餐后不新增住宿；单日行程仍有 `lodging[0]` 并计算 1 晚；
- 第 i 天使用 `start_hotel_id=lodging[i]`；当 i<N-1 时使用 `end_hotel_id=lodging[i+1]`，当 i=N-1 时 `end_hotel_id=null`。若起止酒店不同，表示当天结束时完成换酒店入住，下一天必须从该结束酒店出发；不单列退房、行李和入住耗时；
- `lodging[0]` 依附首日第一个主要点位分布；中间住宿同时考虑前一日最后主要点位和下一日主要点位分布，以减少跨日断裂；
- 预算默认假设为 2 名成人、1 间房。门票、已选餐饮 POI 和公共交通按 2 人计，打车按 1 车计；住宿侧早餐只有在来源明确证明含早或有价格时才计入酒店费/餐饮费，否则早餐费用标记 `unavailable`；自驾费用仅在存在明确里程/停车等确定性规则时估算，否则标记 `unavailable`；
- 打车使用高德驾车路线作为路径事实，但必须标记为 `taxi` 而非 `self_drive`；费用由 BudgetCalculator 按城市可配置的明确计价规则估算，无规则时不得生成价格；
- 页面预算区必须展示人数、房间数、住宿晚数以及公共交通按人、打车按车的计价假设，避免将不同计价单位无说明地相加。
- partial 前缀为 Day1..DayK 且 K<N 时，DayK 仍是原请求中的非末日，必须抵达 `lodging[K]`；partial 预算按实际返回前缀使用的 `lodging[0..K]` 计算 K+1 晚，而不是简单按返回天数 K 计算。

### 3.3 额外要求约束

约束层级固定为：

```text
系统硬约束（真实性、合法来源、时间窗口、必要路线）
  → 已接受的额外要求 ExtraConstraint
  → 三级选址优先级与用户偏好
  → 普通 Planner 建议
```

- 当前测试阶段默认输入者为开发者，并由输入者保证 `free_text_input` 表达的是确定需求；系统不实现面向普通用户的多轮澄清；
- DestinationResearchAgent 使用独立结构化输出将文本按子句解析为封闭判别联合 `ExtraConstraint`；首版只接受 `must_visit_poi`、`dietary_exclusion`、`explicit_time_constraint`、`mobility_constraint`，新增类型必须先更新 schema、执行器和测试；
- 只有类型、约束对象和动作均可明确确定，且通过 Pydantic 与程序规则校验的子句才能标记为 `accepted`；无法归类、对象不明确或存在多种解释的子句标记为 `ignored`，直接丢弃且不得由 Agent 猜测补全；
- 额外要求解析经过有限重试后仍未得到合法结构化结果时，将本次无法确认的全部子句按 `parse_failed` 记入 ignored，并在无额外约束模式下继续规划；这是对技术解析错误分流规则的明确例外。PrimaryDraft、ItineraryDraft、RouteSelection 等核心规划输出解析失败且重试耗尽时仍按 API 错误处理；
- 同一输入包含多个子句时逐条判定，不能因为一个子句模糊而丢弃其他已明确子句；`ignored_extra_requirements` 仅用于测试诊断和结果说明，不参与规划；
- `must_visit_poi` 字段固定为 `query / city / scope=trip`，不绑定具体日期。它必须触发 AmapService 精确检索；无法唯一匹配真实 POI，或最终计划未包含该 POI 时，整个请求 `failed`；
- `dietary_exclusion` 将自然语言归一化为受控排除标签，再由确定性“大众印象规则表”映射到菜系、餐馆类别和名称关键词。例如“不吃辣”归一化为 `avoid_spicy`，直接排除川菜、湘菜、重庆火锅、麻辣烫等高风险类别；即使单店可能提供不辣菜品也不恢复候选，不做菜单级反向推理；
- `explicit_time_constraint` 只接受目标、日期（可选）、时间关系和时间值均明确的要求；缺少任一关键字段则 ignored。accepted 时间约束隐含“目标必须进入行程”：解析器须同时产生关联的 trip-scope `must_visit_poi`，目标无法唯一匹配或最终未安排时整单 failed；
- `dietary_exclusion` 约束所有由系统选择的真实美食级和便餐级 POI，不约束 `lodging_side` 住宿侧早餐；住宿侧早餐由用户在酒店餐厅或附近早餐店自行选择，结果不得声称系统已验证其饮食属性；
- `mobility_constraint` 首版仅接受可由路线距离确定性校验的限制。系统默认单次步行路线不得超过 2km；超过上限只代表 walking 候选不可用，必须继续尝试打车或公共交通，不能仅因此判定约束失败。只有所有非步行替代路线也不可用时，才按必要路线缺失处理；“完整无障碍设施”等无法验证的要求按 unsupported 记入 ignored；
- accepted 约束是硬性用户约束，不得在超窗修复中删除或放宽。任一 accepted 约束无法满足时整个请求 `failed`；系统真实性与时间边界仍优先，但冲突结果也是 failed，而不是返回违反用户约束的 partial。

## 四、数据真实性边界与来源规则

### 4.1 基本原则

1. **事实数据不可生成**：POI、地址、坐标、路线、天气等事实必须来自明确的数据源，LLM 不得补写。
2. **决策数据允许推理**：候选筛选、偏好匹配、分天排序等允许 Agent 判断，但只能操作已有候选 ID。
3. **计算结果由程序负责**：时间轴时刻、预算合计等由确定性代码计算，Agent 不直接输出最终 `TripPlan`。
4. **缺失数据显式降级**：字段缺失时使用 `null` 并标记 `unavailable`，不得使用虚构值维持页面完整。
5. **外部来源不等于绝对正确**：来自高德等外部服务的数据统一标记为 `sourced`，不使用可能造成过度承诺的 `verified`。

### 4.2 数据类型

| 类型 | 含义 | 示例 |
|------|------|------|
| `sourced` | 来自可追溯的外部服务或用户输入 | POI、坐标、路线、天气、用户偏好 |
| `computed` | 由程序按确定性规则计算 | 日期、预算汇总、到达/离开时刻 |
| `estimated` | Agent 或确定性规则给出的估算值 | Agent 建议的游览时长、BudgetCalculator 按规则生成的预估费用 |
| `generated` | LLM 生成的建议性内容 | 景点简介、旅行建议、选择理由、行前须知 |
| `unavailable` | 当前数据源没有提供 | 缺失的票价、评分、营业时间或远期天气 |

### 4.3 字段责任边界

| 数据 | 责任方 | 约束 |
|------|--------|------|
| 首页额外要求原文 | 用户输入 | `free_text_input` 作为来源文本保留，不直接拼接为最终事实或无条件执行 |
| 额外要求结构化判定 | DestinationResearchAgent | 使用 `ExtraConstraintParseResult` 输出逐子句 `accepted / ignored`；accepted 必须为受控类型且字段完整，ignored 不参与后续检索、筛选或规划 |
| 修订约束上下文 | TripPlan + plan_token | TripPlan 携带完整 normalized `accepted_constraints`，不只携带 `constraint_ids`；令牌绑定其内容摘要。首版 revision 不允许新增、删除或放宽 accepted 约束 |
| POI ID、名称、地址、坐标、类型 | AmapService | 必须来自高德返回结果，Agent 不得新增不存在的 POI |
| 景点、酒店、餐厅候选 | AmapService | 负责检索并保留来源 ID；便餐级午餐/晚餐候选必须包含高德返回的具体 POI 名称（含分店名，如有）、地址和坐标，禁止将检索关键词或餐饮类别作为候选实体；无结果时返回空集合 |
| 景点、酒店、餐厅筛选 | DestinationResearchAgent | 只能返回候选集合中已存在的 ID；有效天气进入主要点位筛选时，只能据室内/室外标签替换候选 ID，不得生成新点位。室内/室外若不是高德明确提供的事实字段，只能由受控规则或 Agent 形成 `estimated` 分类，并允许 `unknown`，不得标记为 `sourced` |
| 路线、距离、交通耗时、路径坐标 | RouteService | 底层来自高德路线规划 API，禁止 LLM 估算或补写 |
| 天气 | AmapService | 获取并结构化高德天气数据；仅展示数据源支持日期范围内的预报，超出范围标记 `unavailable`，气候建议不得冒充天气预报 |
| 评分、营业时间、票价 | 外部服务 | 有来源营业时间时，美食级/便餐级午晚餐候选先校验固定餐窗内存在连续 ≥40min 区间；美食级早餐跳过候选餐窗预筛。最终阶段统一校验营业时间完整覆盖实际用餐区间，不满足时替换同级餐厅并重建；缺失时保持 `null/unavailable` 并提示核实 |
| 主要点位分天、排序、时段与删除排名 | ItineraryPlannerAgent | 第一次调用输出 `PrimaryItineraryDraft`；非餐饮主要点包含 `day_part=morning|afternoon`，美食级餐点包含`meal_role=breakfast|lunch|dinner` 且 `day_part=null`。每天至少 1 个主要点位，并包含 provisional 餐次槽位。必打卡点 required 且无删除排名，其他主要点位提供`removal_rank` |
| 连续住宿与便餐筛选 | DestinationResearchAgent | 在 PrimaryDraft 后筛选 `lodging[0..N-1]`，再根据主要点位顺序和起止酒店锚点筛选真实便餐 POI；只能引用候选 ID |
| 完整活动顺序与建议时长 | ItineraryPlannerAgent | 第二次调用输出完整 `ItineraryDraft`，引用 PrimaryDraft、住宿和便餐 ID；酒店、三餐、必打卡点 required，建议时长标记 `estimated` |
| 混合交通路线选择 | ItineraryPlannerAgent | RouteService 先提供步行、打车、地铁/公交及数据源明确支持的旅游专线大巴候选，不包含自驾；Agent 只能返回候选 route ID 和选择理由，不得改写距离、耗时、换乘或路径坐标 |
| 到达/离开时刻、时间冲突检查 | TimelineCalculator / TimelineValidator | 根据真实路线耗时、游览时长和缓冲规则确定性计算与校验 |
| 单项预估费用 | BudgetCalculator | 按 2 名成人、1 间房和实际 `lodging_count` 计算；完整计划 `lodging_count=N`，partial Day1..DayK 且 K<N 时 `lodging_count=K+1`。门票/餐饮/公共交通按人，打车按车；无事实或明确规则时标记 `unavailable` |
| 预算汇总与假设 | BudgetCalculator | 根据有效事实价格与规则估算项确定性计算；同时输出人数、房间数、住宿晚数和交通计价单位。BudgetCalculator 是所有单项预估费用、预算汇总和预算假设字段的唯一提供方 |
| 简介、总体建议、选择理由、行前须知 | ItineraryPlannerAgent | 允许生成，统一标记为 `generated`，不得夹带未经来源支持的事实 |

`PrimaryItineraryDraft` 只允许包含日期、有序主要点位 ID、非餐饮点的 `day_part`、美食级餐点的 `meal_role`、accepted `constraint_ids`、`required`、可选点 `removal_rank`、建议时长、理由和 `provisional_meal_slots`。每个餐次槽位标记 `occupied_by_gourmet` 或 `requires_convenience_meal`；美食级餐点直接占用对应槽位。锚点类型为已有 `POI_ID` 或 `DAY_START / LUNCH_SLOT / DAY_END`：午餐使用前后锚点；无下午点且午餐尚未选定时，`dinner_after_anchor=LUNCH_SLOT`。Stage 9A 先选午餐，将 `LUNCH_SLOT` 解析为真实午餐 POI ID 后再检索晚餐。Stage 8 不得引用尚未选定的酒店或便餐 ID。完整 `ItineraryDraft` 再加入酒店和便餐；每个午晚餐餐次恰好一个真实 POI。

最终数据流固定为：

```text
DestinationResearchAgent（解析 free_text_input → accepted / ignored）
  → TripCoordinator（应用 accepted 约束，必打卡点触发精确检索）
  → AmapService（检索景点、美食级候选与可用天气）
  → DestinationResearchAgent（Stage 18 起结合天气筛选主要点位）
  → ItineraryPlannerAgent(PrimaryItineraryDraft：分天 + 主要点位顺序)
  → AmapService + DestinationResearchAgent（连续住宿记录）
  → AmapService + DestinationResearchAgent（仅为未占用餐次按相邻锚点检索/筛选便餐）
  → ItineraryPlannerAgent(ItineraryDraft：起止酒店 + 全部活动)
  → RouteService（固定方式；混合方式返回多个真实候选）
  → ItineraryPlannerAgent(RouteSelection，仅混合交通)
  → TimelineCalculator / BudgetCalculator → TimelineValidator
  → 不通过时 TripCoordinator 按问题替换路线、替换餐厅、重排 PrimaryDraft 或删减可选主要点，并重建受影响下游（有限循环）
  → TripCoordinator → TripPlan
```

### 4.4 来源元数据

不对每个字段包装 `{value, source}`，避免前后端模型过度复杂。景点、酒店、餐厅、天气和路线等事实实体统一增加轻量来源元数据：

```json
{
  "name": "故宫博物院",
  "address": "北京市东城区景山前街4号",
  "ticket_price": null,
  "data_quality": {
    "source": "amap",
    "source_id": "B000A8UIN8",
    "fetched_at": "2026-06-29T10:00:00+08:00",
    "freshness": "live",
    "estimated_fields": ["visit_duration"],
    "generated_fields": ["description"],
    "unavailable_fields": ["ticket_price"]
  }
}
```

`freshness` 取值限定为：

- `live`：本次请求实时获取；
- `cached`：命中有效期内缓存；
- `stale`：实时调用失败后使用过期缓存，必须向用户提示；
- `not_applicable`：计算值或生成内容不适用新鲜度概念。

### 4.5 防止 Agent 编造的程序约束

1. AmapService 返回带稳定来源 ID 的结构化候选集合。
2. DestinationResearchAgent 只输出候选 ID 和筛选理由，不回写地址、坐标等事实字段。
3. TripCoordinator 根据候选 ID 回填事实实体；Agent 返回未知 ID 时拒绝结果并触发有限重试或降级。
4. ItineraryPlannerAgent 只能使用 TripCoordinator 提供的 ID，先生成 `PrimaryItineraryDraft`，再基于已筛选住宿/便餐生成 `ItineraryDraft`；不得创建新实体或输出最终 `TripPlan`。
5. RouteService 只接收已有点位坐标，并且是路线、距离和交通耗时字段的唯一提供方。
6. TimelineCalculator 根据草案与事实数据确定性生成时间轴和超窗报告，不得修改草案或删除点位；BudgetCalculator 按 2 名成人、1 间房、实际 `lodging_count` 及交通计价单位生成费用。时间和费用字段均不接受 Agent 输出。
7. TimelineValidator 校验未知 POI、住宿连续性、缺失路线、伪造坐标、时间冲突与 accepted 约束；只有 TripCoordinator 可以根据问题执行替换、删点和重算。
8. 只有 TripCoordinator 可以组装并返回最终 `TripPlan`，前端不得消费 Agent 草案。
9. 除 `lodging_side` 早餐外，午餐和晚餐必须引用本次高德餐饮候选集合中的具体 POI ID；仅有“川菜馆”“当地餐厅”等类别文本视为无效必要餐食。
10. `free_text_input` 只能通过校验后的 accepted `ExtraConstraint` 影响规划；ignored 子句不得出现在检索关键词、Prompt 约束、required 点位或最终规划理由中。
11. 必打卡约束只能绑定 AmapService 返回的唯一真实 POI ID；无法唯一匹配或最终未打卡时整个请求 failed。accepted `explicit_time_constraint` 同时产生关联的 required 目标。`dietary_exclusion` 必须通过固定规则表在美食级/便餐级候选过滤、完整草案和最终校验三处保持一致，但不校验 `lodging_side` 住宿侧早餐。

### 4.6 数据降级规则

```text
实时数据成功       → sourced / live
实时失败且缓存有效  → sourced / cached
仅有过期缓存       → sourced / stale，degraded=true，并展示降级提示
完整范围经跨日重排与有限修复后，第 K+1 日仍无法形成完整日 → 仅返回连续前缀 Day1..DayK 并标记 partial
首日无法形成完整日 → failed
一个完整可执行日都没有 → failed
任一 accepted 额外要求无法满足 → 整个请求 failed，不返回 partial
```

坐标修复只能使用高德地理编码结果或带来源的有效缓存。Fallback 不再生成“某城市景点1”、北京默认坐标、算法平移坐标或其他虚构事实。

计划完整性状态定义为：

- `complete`：请求范围内的每日计划均可执行，每日至少包含 1 个主要点位，所有已安排点位具有合法来源，所需路线和时间轴完整；美食偏好开启时包含 1～N 个美食级餐点且每天最多 1 个；票价、评分等可选事实缺失不影响完整性；
- `partial`：只能在完整日期范围已经过跨日重排主要点位、餐次、住宿及有限修复后生成，并返回从请求首日开始、日期连续的最长完整可执行前缀 `Day1..DayK`；遇到首个失败日后立即截断，后续日期不得跳跃返回。若 K<N，DayK 按非末日规则抵达结束酒店；
- `failed`：无法形成满足真实性、基本可执行约束、accepted 额外要求及美食偏好数量硬规则的正式结果，不返回伪造兜底数据。

因此，只有首日开始至少存在 1 个连续完整日才允许返回 `partial`；首日失败时必须 `failed`。可选票价、评分、营业时间等缺失不影响完整日判定。

`partial` 只用于外部数据或必要安排导致部分日期无法形成完整日的情况，不能用于绕过 accepted 额外要求。必打卡 POI 无法唯一匹配、最终未被安排，或其他 accepted 约束无法满足时，整个请求必须 `failed`。

美食偏好开启时，最终 `complete` 结果必须包含 1～N 个美食级餐点，`partial` 前缀必须包含 1～K 个且每天最多 1 个。若初始可执行前缀中没有美食级餐点，Coordinator 必须先尝试将真实美食级餐点跨日重排到前缀并重建下游；仍无法满足时整个请求 `failed`，不得返回不含美食级餐点的 partial。

上游返回合法空候选、必要 POI/路线确实不存在或修复后时间轴仍不可行，属于业务不可行，可按上述规则形成 `partial/failed`。配置错误、网络超时、高德官方接口/MCP 异常或核心规划输出解析异常在有限重试且无可用缓存后仍无法提供可信规划输入时，返回分级 API 错误；额外要求解析失败按 3.3 节的明确例外进入 ignored。成功使用 stale 缓存时正常返回计划并设置 `degraded=true`。

`degraded` 与 `plan_status` 相互独立：使用 stale 缓存或其他降级数据路径时设为 `true`。因此结果可以是 `complete + degraded` 或 `partial + degraded`，但不能将 `partial` 作为 `complete` 展示。

### 4.7 验收规则

1. 所有景点、酒店和餐厅 ID 均能在本次候选集合或带来源的缓存中找到。
2. 所有路线段均由 RouteService 返回，不存在 LLM 生成的距离、耗时或路径坐标。
3. 事实字段缺失时为 `null`，并出现在 `unavailable_fields` 中。
4. BudgetCalculator 输出的预算合计与 2 名成人、1 间房、实际 `lodging_count` 以及各交通计价单位下的有效明细一致；完整计划为 N 晚，partial 前缀按 K+1 晚，前端展示同一数值。
5. 超出天气数据支持范围的日期不展示伪预报。
6. `stale` 数据、部分计划和其他降级结果均有明确状态与用户提示。
7. `complete / partial / failed` 与 `degraded` 的组合符合定义；`partial` 中每个返回日均具备酒店、三餐、必要路线和完整时间轴，前端不得将部分或失败结果展示为完整成功。
8. 每个完整日的午餐和晚餐均对应具体高德餐饮 POI，并出现在有序活动、路线段和时间轴中；泛化餐馆类别不得通过验收。
9. 明确的额外要求优先于偏好和普通规划建议；必打卡点不会进入删除排名，固定大众印象规则表产生的饮食排除不会在后续阶段失效。
10. 模糊子句被记录到 `ignored_extra_requirements` 且对候选和计划无影响；同一输入中的其他明确子句仍正常生效。
11. `lodging_count=len(lodging)`；complete 为 N，partial Day1..DayK 且 K<N 时为 K+1。`lodging[i]` 与相邻返回日期的 `start_hotel_id / end_hotel_id` 严格连续；非末日最后活动到结束酒店存在真实路线。
12. TimelineCalculator 只返回时间轴和超窗问题，不直接改变活动；所有替换、删点和重算均可追溯到 TripCoordinator。
13. 午餐、普通日晚餐和末日晚餐分别完整落在固定窗口；非餐饮主要点不与午餐重叠。美食级/便餐级午晚餐候选在对应窗口内须存在连续 ≥40min 可用区间；美食级早餐不做候选餐窗预筛。所有最终实际用餐区间均须被有来源营业时间完整覆盖；缺失时标记 unavailable 并提示核实。
14. 交通偏好方式不可用或不合格时自动查询其他真实方式；walking 仅在真实路线 ≤2km 时可选，>2km 强制尝试打车/公共交通。回退候选按耗时、距离、route ID 稳定选择；accepted 移动约束不得放宽。
15. 生成 partial 前必须完成全日期跨日重排和有限修复；日期类型以 `day.date == request.end_date` 判定，不以返回数组末项判定。partial K<N 的所有返回日均按非末日规则，DayK 抵达 `lodging[K]`；住宿链和预算按实际前缀重算。
16. 每个完整日至少包含 1 个主要点位；美食级餐点按 `meal_role` 进入对应正式餐次并可计为主要点位，同一餐次不得再安排便餐。
17. accepted `explicit_time_constraint` 的目标同时为 required；饮食排除覆盖系统选择的真实餐饮 POI，但不覆盖用户自行选择的住宿侧早餐。
18. 业务不可行与技术故障分流：前者进入 `partial/failed`，后者在无可信输入时返回稳定 API 错误；stale 缓存成功时使用 `degraded=true`。
19. 美食偏好开启时，complete 含 1～N 个美食级餐点，partial 含 1～K 个且每天最多 1 个；无法通过跨日重排使前缀满足下限时整单 failed。
20. 额外要求解析失败且重试耗尽时按 `parse_failed` ignored 继续规划；核心 PrimaryDraft、ItineraryDraft、RouteSelection 解析失败且重试耗尽时返回稳定 API 错误。
21. 有效天气先参与 DestinationResearchAgent 的室内外主要点筛选/替换，再由 ItineraryPlannerAgent 分天排序；点位集合、日期或顺序变化后重建全部下游，超预报范围不产生天气决策。
22. 无下午点且午餐尚未选定时，`dinner_after_anchor=LUNCH_SLOT`；Stage 9A 先选择午餐并解析为真实 POI ID，再检索晚餐，不得用 `DAY_END` 代替午餐锚点。
23. revision 上下文携带并签名完整 normalized `accepted_constraints`；首版支持更换美食级餐点、更换便餐和修改 `meal_role`，但不得修改 accepted 约束，删除必打卡等冲突指令不执行。
24. revision 与 accepted 约束冲突时，当前 TripPlan 和已采纳对话均不改变；输入文字保留，文字下显示“与额外要求冲突，不予采纳”，输入框整体红边，用户修改任意内容时立即清除提示并恢复边框。

## 五、高德 API 依赖与验证约束

`RouteService` 不是高德提供的独立服务，而是项目内部封装。它复用 `AmapService` 接入高德路线规划能力，并向 TripCoordinator 和前端提供统一的业务数据结构。AmapService 是唯一外部高德数据边界：优先验证 MCP 通道，验证不通过时允许直接调用高德官方 HTTP API，RouteService、Agent 和前端不得分别绕过 AmapService 自行选择通道。

单次行程的 API 调用量取决于偏好数量、候选数量、旅行天数、路线段数量与缓存命中情况，不能以固定次数作为设计前提。高德官方配额、计费规则和字段能力可能调整，也不能在 02 文档中承诺固定日配额、可支持规划次数或所有调用均免费。

03 文档必须安排技术验证，至少确认：

- 目标版本 `amap-mcp-server` 的实际工具列表与返回结构；
- POI 来源 ID、餐饮营业时间及其缺失/分段表达、路线距离、耗时和路径坐标是否稳定可用；
- 同步串行调用下 MCP 实例的连续调用稳定性与资源释放行为；
- 当 MCP 字段缺失、格式不稳定或资源释放不可靠时，高德官方 HTTP API 对等接口的字段、错误、限流和鉴权行为；
- 目标高德 Key 的实时官方配额、限流和计费规则；
- 典型 1～3 日行程的实际调用量与失败率。

缓存、限流和降级方案必须以验证结果为依据，不得依赖未经确认的固定数字。本阶段不实施查询并发；除配额外，还需关注 `amap-mcp-server` 第三方中间层的兼容性与稳定性。若验证不通过，切换为高德官方 HTTP API 是既定技术回退，不属于数据降级，也不得因此把结果标记为 `degraded`；只有使用 stale 数据等质量下降路径时才设置 `degraded=true`。

**对策：AmapService 缓存降级**

```
search_poi("故宫", "北京")
  ├─ 查有效缓存 → 命中 ✅
  ├─ 未命中 → 已选定实时通道（MCP 或高德官方 HTTP API）→ 解析 + 写缓存
  └─ 实时通道异常 → stale 缓存兜底 ⚠️ / 触发分级错误
```

当前单机、单进程、单实例范围允许采用进程内缓存，不要求引入 Redis 或其他分布式缓存。各类数据 TTL、容量和清理策略由 03 文档根据技术验证结果确定；进程重启后缓存可以丢失，PDF 历史目录不属于缓存，必须使用可持久化固定磁盘。

### 6.3 设计验收标准

1. **不退化**：每个景点/美食目的地保留现有信息区域；评分、费用等事实缺失时显示“暂无数据”，不得为维持信息密度而生成虚构值。
2. **可执行**：用户能依次读出一条路径——几点出发、怎么去、多久到、玩多久/吃多久、中间有没有缓冲。
3. **可对比**：用户手动同时展开多天后，不同天的时间轴可纵向对齐对比。
4. **可联动**：hover 时间轴某段 → 地图对应路线高亮，反之亦然。
5. **两级餐饮**：美食偏好开启时，complete 有 1～N 个、partial 有 1～K 个美食级餐点且每天最多 1 个；美食目的地使用景点级卡片，但必须按 `meal_role` 进入对应正式餐次并替代同餐次便餐。未安排美食级早餐时显示住宿侧早餐及政策核实提示；所有实际在外用餐均预留 ≥40min。
6. **可追溯**：RouteService 给出的每段路线都可用同起终点与策略在高德路线规划 API 中复核（可查得到），禁止使用 LLM 编造路线；不强制要求路线来源持久化。
7. **设备范围**：本阶段仅验收 PC 端显示与交互完整性，不纳入移动端/平板适配要求。
8. **行程时长**：非末日住宿侧早餐方案不超过 12 小时，美食级早餐方案不超过 13 小时，并抵达 `end_hotel_id`；末日分别不超过 9 小时和 10 小时，不设置结束酒店，晚餐须在 18:00 前结束。所有上限均不得视为必须填满的额度。
9. **独立缓冲**：每段真实交通之外至少保留 30 分钟 `buffer`，交通耗时不得替代缓冲，缓冲也不得预先安排其他活动。
10. **早餐规则**：住宿侧早餐按规划约定在 09:00 前完成并从酒店直奔首个主要点位，是否含早及具体地点须提示用户核实；美食级早餐必须 08:00 出发，先完成早餐点的交通、缓冲和用餐，再前往主要点位。
11. **预算假设**：预算区明确展示 2 名成人、1 间房、实际 `lodging_count`，以及公共交通按人、打车按车的计算口径；完整计划显示 N 晚，partial 显示 K+1 晚。
12. **PDF 完整性**：导出按钮不依赖当前展开状态；点击后自动在导出容器中展开全部天级卡片，导出完成后恢复用户原有展开状态。
13. **餐次真实性与参与规划**：除住宿侧早餐外，每日午餐和晚餐均恰好对应一个高德真实美食级或便餐级 POI，显示具体店名/分店名并具有地址、坐标和来源 ID；二者出现在活动顺序、相邻路线和时间轴中，不得重复安排或仅显示泛化餐饮文字。
14. **住宿连续性**：非末日同时展示出发酒店和结束酒店；若二者不同，时间轴和地图必须显示最后活动到新酒店的路线，下一日从该新酒店开始。
15. **固定用餐窗口**：午餐完整落在 12:00—14:00，普通日晚餐完整落在 18:00—20:00，末日晚餐完整落在 17:00—18:00；上午/下午点位与午餐边界一致。
16. **步行回退**：单段 walking 路线仅在 ≤2km 时展示为可选；超过 2km 时页面展示实际选定的打车或公共交通路线，不显示“步行超限导致失败”。
17. **交通偏好回退**：所选交通方式不可用时展示按真实耗时、距离和 route ID 稳定选出的替代方式，并明确实际采用的交通方式；accepted 移动约束不得被回退绕过。
18. **营业时间提示**：午晚餐候选在固定餐窗内须有连续 ≥40min 可用区间；美食级早餐不做固定餐窗预筛。所有最终用餐区间均须被有来源营业时间覆盖；缺失时显示“营业时间待核实”。
19. **对话约束冲突态**：冲突指令不执行且不进入已采纳对话；保留用户输入，在其下显示“与额外要求冲突，不予采纳”，输入框整体边框变红。用户修改内容时立即移除提示并恢复正常边框，重新提交后再由后端判定。
