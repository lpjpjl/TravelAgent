## 二、优化方向汇总

### P0 — 核心能力提升（6 项）

| 编号 | 优化项 | 问题 | 说明 |
|------|--------|------|------|
| P0-1 | **多偏好并行搜索** | 1 | 遍历所有偏好标签，并行发起搜索。当偏好包含"美食"时，自动触发美食目的地搜索（与景点等权），其余偏好正常处理 |
| P0-2 | **标准化 Tool Calling** | 2 | 使用框架标准的 function calling 替代 `[TOOL_CALL:...]` 文本格式 |
| P0-3 | **Agent 并行执行** | 3 | 景点/天气/酒店 Agent 用 asyncio 并行运行 |
| P0-4 | **结构化输出** | 5 | PlannerAgent 使用 JSON mode / structured output |
| P0-5 | **行程深度关联化** | 22 | Coordinator 统一调度各专家 Agent 输出并做关联校验（景点↔餐饮↔交通↔酒店闭环）。餐饮检索统一由 MealAgent 执行，并采用**两级模型**：美食偏好开启时，餐饮分为美食级（1-3 个，提升为景点级）与便餐级（就近依附景点）；美食偏好关闭时，餐饮全为便餐级并就近依附景点。无论美食级/便餐级，均参与时间轴决策并预留充足就餐时段（便餐每顿 ≥40min）。高松弛度原则（日 ≤ 8h、间隔 ≥ 30min）。天气驱动、行前须知、时间约束等子项由 E-5/E-6/E-7 分阶段补充 |
| P0-6 | **可执行时间轴信息** | 22 | 输出 Day Timeline（到达/离开时间、路程耗时、缓冲时间、当日总时长）。前端以**复合时间轴卡片**展示：左侧时间轴轨道 + 右侧点位卡片（保留实景图、名称、地址、游览/用餐时长、简介、评分、费用等现有信息），不允许退化为纯时间线 |

## 四、高德 API 依赖分析

一次 3 天行程 ≈ **25 次 API 调用**。免费版日配额 **5000 次**，可支撑约 200 次规划。所有 API 均在免费版范围内。

**风险不在配额，在 `amap-mcp-server` 第三方中间层的稳定性。**

**对策：AmapService 缓存降级**

```
search_poi("故宫", "北京")
  ├─ 查本地缓存（JSON, TTL 24h）→ 命中 ✅
  ├─ 未命中 → MCP → 高德 → 解析 + 写缓存
  └─ MCP 异常 → stale 缓存兜底 ⚠️ / 触发分级降级
```

| 缓存参数 | 值 |
|----------|-----|
| POI TTL | 24h |
| 天气 TTL | 2h |
| 路线 TTL | 24h |
| 存储 | `backend/data/cache/` (JSON) |

### 5.3 设计验收标准

1. **不退化**：每个景点/美食目的地的图片、名称、地址、时长、评分、费用、简介均可见，不低于当前信息密度。
2. **可执行**：用户能依次读出一条路径——几点出发、怎么去、多久到、玩多久/吃多久、中间有没有缓冲。
3. **可对比**：用户手动同时展开多天后，不同天的时间轴可纵向对齐对比。
4. **可联动**：hover 时间轴某段 → 地图对应路线高亮，反之亦然。
5. **两级餐饮**：美食偏好开启时，美食目的地 = 景点级卡片（同模板）；其余三餐 = Day 底部便餐条，但每顿时间轴预留 ≥40min。
6. **可追溯**：Agent 给出的每段路线都可用同起终点与策略在高德路线规划 API 中复核（可查得到），禁止使用 LLM 编造路线；不强制要求路线来源持久化。
7. **设备范围**：本阶段仅验收 PC 端显示与交互完整性，不纳入移动端/平板适配要求。
