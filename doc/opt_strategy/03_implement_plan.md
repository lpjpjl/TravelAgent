# 旅行 Agent 优化实施计划（初版）

> 依据：`02_opt_plan.md`
>
> 文档状态：初版，供后续评审与迭代；本文定义开发顺序，不改变 02 文档中的产品优先级。

---

## 一、实施目标与编排原则

本轮实施的核心目标是：先消除虚构数据和脆弱工具调用，再建立可执行、可追溯的结构化行程，随后补齐可靠性与用户体验，最后接入对话式迭代和低优先级工程治理。

### 1.1 排期规则

1. **小步快跑**：每个 Stage 只交付一个主要能力；一个 Stage 内不得同时重写 Agent、服务层和前端主流程。
2. **阶段可验收**：每个 Stage 都必须有自动化测试、明确输入输出和独立验收条件；未通过退出条件不得进入依赖它的 Stage。
3. **先契约后实现**：先确定 schema、状态和错误语义，再替换生产调用链，避免前后端反复返工。
4. **事实链优先**：外部事实必须来自 AmapService 或可追溯缓存；Agent 仅做候选筛选与规划决策。
5. **渐进迁移**：重构期间保留旧 `/trip/plan` API 的兼容层；新链路稳定后再删除旧 Agent、旧 schema 和虚构 fallback。
6. **不按 P 编号机械开发**：P 级代表价值，不代表依赖顺序。P0-9 因涉及会话、重规划和 Agent 架构调整，按要求整体放在 **P2 全部完成之后、P3 开始之前**。
7. **PC 优先**：当前阶段不引入移动端和平板适配目标。
8. **同步优先**：景点、天气、酒店、餐饮和路线等外部查询按依赖顺序同步串行执行；本计划不实施查询并发优化。

### 1.2 Stage 完成定义（Definition of Done）

每个 Stage 均须满足以下通用条件：

- 代码通过后端单元/契约测试和前端类型检查、构建；
- 新增关键分支有测试，不以人工页面观察代替数据正确性验证；
- API/schema 变更同步更新前端类型和接口文档；
- 外部调用可观测，日志包含 `request_id`、阶段名、耗时、结果状态，不打印 Key 和完整敏感请求；
- 降级、失败和空结果均有显式行为，不使用虚构数据兜底；
- 保留可回滚点，且未破坏上一个 Stage 已通过的验收用例。

## 二、目标架构与迁移边界

### 2.1 目标调用链

```text
Trip API
  → TripCoordinator（普通编排代码）
    → AmapService（同步独立检索 POI / 天气 / 路线底层数据）
    → DestinationResearchAgent（一级主要点位 → 二级酒店 → 三级便餐）
    → ItineraryPlannerAgent（仅返回 ItineraryDraft）
    → RouteService（固定方式；混合方式返回多个真实候选）
    → ItineraryPlannerAgent（仅混合交通时选择真实 route ID）
    → TimelineCalculator / BudgetCalculator（确定性计算）
    → TimelineValidator（真实性与可执行性校验）
    → 校验失败时删点并重算受影响路线（有限循环）
  → TripPlan（唯一正式结果）
```

前端只能消费 `TripPlan`，不得消费 Agent 原始输出或 `ItineraryDraft`。

### 2.2 代码目录建议

目录名称可在实现时微调，但职责不得重新混合：

```text
backend/app/
  agents/
    destination_research_agent.py
    itinerary_planner_agent.py
  coordinators/trip_coordinator.py
  models/
    source.py
    candidate.py
    itinerary.py
    trip.py
  services/
    amap_service.py
    route_service.py
    timeline_calculator.py
    budget_calculator.py
    timeline_validator.py
    cache_service.py
    report_archive_service.py
  api/routes/
    trip.py
    trip_stream.py
    report_history.py
    trip_revision.py
```

### 2.3 迁移期间的硬约束

- `MultiAgentTripPlanner` 仅作为临时兼容入口，不继续堆叠新能力；
- 禁止新增 `[TOOL_CALL:...]` Prompt；
- 禁止 Agent 输出或改写地址、坐标、距离、路线耗时、事实价格和预算合计；
- 新链路不得调用 `_create_fallback_plan` 一类虚构兜底；
- schema 新增字段优先采用兼容性扩展；需要破坏性变更时使用 API 版本或迁移适配器；
- 当前结构化 TripPlan 只保存在 Pinia，并用 sessionStorage 做当前标签页临时恢复；不引入结构化行程数据库；
- 只有用户执行“导出行程为 PDF”后产生的 PDF 才进入历史：同一 PDF 既由浏览器下载到本地，也上传到后端固定磁盘目录；历史 PDF 不支持恢复为结构化行程或继续对话；
- 删除现有手动编辑入口；Stage 14 起行程页保持只读，直到 Stage 32 上线对话式迭代；
- 新链路在 Stage 5～15 仅通过功能开关或 shadow 模式验证，Stage 16 完成错误、状态与虚构 fallback 治理后才切为默认；
- 旧链路的删除必须晚于新链路全量验收，不在中间 Stage 提前清理。

## 四、详细实施阶段

### Stage 1：高德 MCP 技术验证（独立难点 Stage）

**目标**：消除 02 文档明确列出的外部能力不确定性，为后续设计提供实测依据。

**范围**：

- 核验目标版本 `amap-mcp-server` 的工具名、入参、返回内容层级和异常形态；
- 核验 POI ID、经纬度、天气日期、路线距离、耗时、polyline 是否稳定可取；
- 分别验证步行、驾车、公交路线；记录地址调用与坐标调用的差异；
- 验证同步串行连续调用下 MCP 实例的稳定性、超时行为和资源释放；
- 查询目标 Key 当前官方配额、限流和计费规则；
- 对典型 1～3 日请求测量调用数、延迟与失败率，产出决策记录。

**交付物**：`doc/opt_strategy/decisions/amap_mcp_validation.md`、脱敏 fixture、工具能力矩阵。

**退出条件**：能回答“用什么工具、返回什么、串行调用是否稳定、失败如何识别、配额如何约束”；否则阻断 Stage 3、4、10A、10B、17，不以猜测继续实现。

### Stage 2：核心领域契约

**目标**：先固定各层边界，再迁移调用链。

**范围**：

- 定义 `DataQuality`、`freshness`、`PlanStatus`、`degraded` 和缺失项结构；
- 将旅行天数限制为 1～3 天、偏好限制为最多 3 个；校验 `travel_days == end_date - start_date + 1`；
- 将交通方式、住宿类型和偏好定义为前后端一致的受控枚举，拒绝未知值；
- 定义结构化候选、空间锚点、三级规划优先级、筛选结果、`ItineraryDraft`、`RouteCandidate`、`RouteSelection`、`RouteSegment`、`TimelineEvent`、`TripPlan`；
- `ItineraryDraft` 必须包含点位优先级和删除顺序，所有 Agent 输出模型统一设置 Pydantic `extra="forbid"`；
- 定义配置错误、超时、上游错误、解析错误、校验错误及稳定错误码；
- 将可缺失事实字段改为 `Optional`，缺失值为 `null`，不得用 `0` 或空字符串伪装真实值；
- 写 schema 正反例和序列化兼容测试。

**不做**：不接 MCP、不改前端页面。

**退出条件**：1～3 天、日期一致性、最多 3 个偏好、全部枚举和 02 文档 4.1～4.7 的真实性/来源/状态组合均能由模型表达并通过正反例测试。

### Stage 3：AmapService POI 与地理编码结构化解析

**目标**：让景点、酒店、餐厅和坐标进入统一的真实数据入口。

**范围**：

- 将 POI 搜索与地理编码原始文本解析为 Pydantic 模型；
- 保留 `source_id`、抓取时间、原始类型和数据质量元数据；
- 对空结果、字段缺失、非法坐标、返回格式漂移显式报错或返回空集合；
- 解析逻辑用 Stage 1 fixture 做契约测试。

**不做**：不实现筛选策略、不实现缓存。

**退出条件**：不再返回 TODO 空列表；任何输出 POI 都有合法来源 ID 和坐标，格式漂移不会静默产生错误实体。

### Stage 4：AmapService 天气与路线结构化解析

**目标**：补齐其余确定性外部事实。

**范围**：

- 天气解析为带日期和来源的结构化数据，超出预报范围标记 `unavailable`；
- 路线解析为距离、耗时、方式、路径坐标和策略元数据；
- 路线失败不得返回空字典冒充成功；
- 建立天气与三类路线 fixture 契约测试。

**退出条件**：AmapService 核心数据调用不再存在未解析 TODO；路线字段可用同起终点和策略复核。

### Stage 5：标准化数据调用链 shadow 接入

**目标**：移除 Agent Prompt 中脆弱的文本工具调用。

**范围**：

- 新建最小 `TripCoordinator` 骨架，由其直接调用 AmapService；
- 景点、酒店、天气数据以结构化对象传入后续处理；
- 新链路通过独立测试入口、功能开关或 shadow 模式运行，不在本 Stage 替换默认 `/trip/plan`；
- 新链路删除 `[TOOL_CALL:...]` 依赖，旧链路仅作为 Stage 16 前的临时兼容入口，不再扩展能力。

**不做**：不在此 Stage 完成新 Planner、路线闭环或时间轴。

**退出条件**：新链路不再要求 LLM 生成工具调用字符串，调用失败能传播为显式错误；默认生产入口仍可回滚到旧实现。

### Stage 6：多偏好完整检索

**目标**：所有用户偏好都进入候选检索，不再只取第一项。

**范围**：

- 建立偏好到检索关键词的可测试映射；
- 请求层最多接受 3 个偏好；按用户选择顺序同步串行检索并按来源 ID 去重；
- 为每个偏好设置候选上限和合并后的总候选上限，避免 Agent 输入无界增长；
- 保留每个候选命中的偏好标签，供筛选 Agent 使用；
- 无偏好时使用明确的通用景点策略。

**退出条件**：多偏好样本中每个标签都有检索记录；合并结果稳定、可去重、可追溯。

### Stage 7：三级候选池与空间锚点

**目标**：建立独立检索、分级关联的景点/美食、酒店和便餐候选池。

**范围**：

- 景点、美食级餐点候选、酒店和便餐候选分别通过 AmapService 同步串行独立检索；
- 第一优先级候选为景点和美食级餐点；第二优先级酒店候选必须计算其与第一优先级候选的真实空间关系；
- 第三优先级便餐候选必须携带其与主要点位及酒店的 `anchor_poi_ids` 和确定性距离，供后续筛选与规划使用；
- 无论是否选择美食偏好都检索餐饮候选，美食偏好决定是否将部分餐厅提升为第一优先级；
- 未安排美食级早餐时，不额外推荐外部便餐早餐；
- 餐厅事实缺失遵循 `null + unavailable_fields`。

**不做**：不让服务层替 Agent 做最终餐厅选择。

**退出条件**：三类候选均有来源 ID；酒店和便餐候选具有可验证的空间锚点；美食偏好开/关两组契约测试通过。

### Stage 8：DestinationResearchAgent 分级结构化筛选

**目标**：建立唯一候选筛选 Agent，并约束其只能引用已有 ID。

**范围**：

- 按第一优先级主要点位、第二优先级酒店、第三优先级便餐的顺序筛选，后一级必须引用前一级锚点；
- 统一使用 JSON mode/`response_format` 适配器获取 JSON 字符串，再由 `extra="forbid"` 的 Pydantic 模型校验，不假设 HelloAgents 原生返回模型；
- 温度固定为 0；输出仅含候选 ID、分类和筛选理由；
- 未知 ID、额外字段或 schema 错误最多重试 2 次，仍失败则显式降级；
- 美食偏好开启时选出 1～3 个美食级候选及便餐级候选，关闭时仅选便餐级。

**退出条件**：Agent 无法向正式结果注入候选集外的 POI；结构校验、未知 ID 和重试上限测试通过。

### Stage 9：ItineraryPlannerAgent 结构化草案

**目标**：Planner 只负责决策，不再直接生成最终 `TripPlan`。

**范围**：

- 输入为分级筛选后的候选、日期、偏好和可用天气；
- 输出严格限制为日期、有序点位 ID、用途、规划优先级、删除顺序、建议时长、规划说明；
- 使用 JSON mode/`response_format` 适配器、`extra="forbid"` Pydantic 校验和最多 2 次有限重试；
- 温度固定为 0.3；
- 每天按三餐规划，并表达酒店早餐/美食级早餐决策；
- 禁止输出事实地址、坐标、路线、最终时刻、事实/估算费用和预算合计。

**退出条件**：`ItineraryDraft` Pydantic 校验通过；非法字段、未知 ID、漏日期和重复安排均有测试。

### Stage 10A：RouteService 固定交通基础版（独立难点 Stage）

**目标**：提供统一、可追溯的相邻点位真实路段。

**范围**：

- 实现 `TripCoordinator → RouteService → AmapService` 调用；
- 步行、公共交通和自驾偏好只查询对应方式；RouteService 只做参数适配和统一返回，不自行估算路线；
- 输入只接受已有点位坐标，输出距离、耗时、方式、polyline、来源和策略；
- 对酒店→首点、点位间、末点→酒店逐段计算；
- 任一必要路线缺失时返回可识别的校验输入，不生成直线兜底。

**退出条件**：典型单日所有相邻段可追溯；路线失败和缺失 polyline 的处理规则固定并有测试。

### Stage 10B：混合交通真实候选选择（独立难点 Stage）

**目标**：在不生成路线事实的前提下，为“混合交通”逐路段选择更方便的真实路线。

**范围**：

- RouteService 对每个相邻路段同步串行查询高德实际支持的步行、公共交通和驾车路线候选；
- 候选统一包含 route ID、距离、耗时、换乘信息、polyline、来源和策略；不可用方式不进入候选集合；
- ItineraryPlannerAgent 使用独立的 `RouteSelection` JSON mode 调用，只能返回已有 route ID 和选择理由；
- `RouteSelection` 使用 `extra="forbid"` 校验、候选 ID 校验和最多 2 次有限重试；
- Agent 无有效选择时不编造路线，由 Coordinator 按明确的最短耗时确定性规则选择，或在无候选时进入 partial/failed。

**退出条件**：固定输入下每段选择均引用真实 route ID；Agent 不可能改写路线事实；缺候选、非法 ID 和重试耗尽测试通过。

### Stage 11：TimelineCalculator

**目标**：用确定性代码生成符合 02 文档约束的完整时间轴。

**范围**：

- 生成出发、交通、独立缓冲、到达、活动/用餐、离开、返店事件；
- 每段真实交通外强制加入至少 30 分钟独立 `buffer`；
- 所有在外用餐至少 40 分钟；
- 酒店早餐 09:00 出发，美食级早餐 08:00 出发；
- 非末日最晚 21:00，末日最晚 18:00；单日游按末日规则；
- 超窗时只移除低优先级点位，不压缩真实交通、用餐和缓冲。

**不做**：不实现前端时间轴。

**退出条件**：02 文档 3.1 的四种时间窗口、首末日、单日游和按删除顺序输出待移除点位均有边界测试；TimelineCalculator 本身不复用删点前路段。

### Stage 12A：BudgetCalculator

**目标**：让所有费用明细与汇总只由确定性代码产生。

**范围**：

- BudgetCalculator 成为单项预估费用和预算汇总的唯一提供方；
- 仅使用事实价格或有明确依据的规则估算，否则为 `unavailable`；
- 记录每个估算项的依据和 `estimated_fields`，不允许 Agent 费用进入正式结果。

**退出条件**：预算明细与合计恒等；事实价格、规则估算、无规则三类测试通过。

### Stage 12B：TimelineValidator

**目标**：建立正式结果组装前的真实性与可执行性质量闸门。

**范围**：

- 校验未知 POI、非法/修复失败坐标、缺失路线和 Agent 越界字段；
- 校验时间冲突、餐饮时长、独立缓冲、每日窗口和首末日边界；
- 校验失败返回结构化问题列表，不允许作为 `complete` 返回；
- 区分可形成 partial 的缺失和必须 failed 的关键错误。

**退出条件**：02 文档 4.5～4.7 的验收规则形成自动化测试，且每种失败均有稳定问题码。

### Stage 13：TripCoordinator 删点重算闭环与最终组装

**目标**：完成景点、酒店、餐饮、路线、时间、预算闭环，并由唯一入口组装 `TripPlan`。

**范围**：

- 按固定调用链编排各 Agent 和确定性服务；
- 按来源 ID 回填事实字段，组装唯一正式结果；
- 为便餐选择提供与前后点位地理相关的候选上下文；
- 执行有上限的修复循环：路线/时间轴/校验失败 → 按草案删除顺序移除低优先级点位 → 重新校验酒店与便餐空间锚点并按需重选 → 重新查询新相邻点位路线 → 重算时间轴和预算 → 再校验；
- 修复循环达到上限仍不通过时返回符合定义的 partial/failed，不压缩真实交通、最低用餐时间或缓冲；
- 根据校验结果生成 `complete / partial / failed`，独立计算 `degraded`；
- 保留旧 API 兼容响应适配，前端切换前不删除旧字段。

**退出条件**：1 日、3 日、美食开/关、候选不足、路线失败、删点产生新相邻路段、删点导致锚点失效八类端到端契约用例通过；失效酒店/便餐会重选，新相邻路段必定重新查询，正式结果中无 Agent 生成的事实字段。

### Stage 14：复合时间轴卡片与移除手动编辑

**目标**：按 02 文档第六章展示可执行时间轴，同时保留现有点位信息密度。

**范围**：

- 左侧时间轨道 + 右侧点位卡片；
- 展示出发/到达/离开、交通、距离、耗时、独立缓冲、用餐和当日在外时长；
- 景点和美食级餐厅使用同级卡片；便餐位于 Day 底部餐饮条；
- 事实缺失显示“暂无数据”，不使用 `0` 伪装；
- 删除地址、时长、顺序和删点等所有前端手动编辑入口；对话式迭代上线前行程页保持只读；
- 本 Stage 只预留地图联动事件，真实地图联动由 Stage 19 实现。

**退出条件**：02 文档 6.3 中除地图联动外的验收项通过；代码中不存在直接修改 TripPlan 事实、路线、时间和预算字段的 UI 入口；PC 主流宽度下无信息截断和布局退化。

### Stage 15：分级错误处理

**目标**：让调用方和用户能区分可恢复与不可恢复故障。

**范围**：

- 映射配置错误、网络超时、上游限流、MCP 异常、LLM 异常、解析异常、校验失败；
- API 返回稳定错误码、可展示消息和 `request_id`；
- 前端按类型显示重试、检查配置或稍后再试，不暴露内部堆栈。

**退出条件**：每类错误均有 API 测试和对应 UI 状态，500 不再承载所有语义。

### Stage 16：分级 Fallback、结果状态与新链路切换

**目标**：彻底移除虚构兜底，正确表达完整性与降级。

**范围**：

- 坐标仅允许高德地理编码或带来源缓存修复；
- 删除北京默认坐标、算法平移、虚构名称和 LLM 补事实路径；
- 正确处理 `complete / partial / failed × degraded` 组合；
- partial 明确列出缺失项，failed 不返回伪造计划主体；
- 前端不得把 partial/failed 显示为完整成功。
- Gate B 全部通过后，将新 TripCoordinator 链路切为默认 `/trip/plan`；旧链路仅保留可快速关闭新能力的短期回滚开关；
- 切换后生产请求不得再进入旧 Planner 的虚构 fallback。

**退出条件**：注入空 POI、坐标失败、路线失败和 Agent 非法 ID 时均不会生成虚构事实；默认生产入口使用新链路，并完成一次可验证回滚演练。

### Stage 17：请求缓存与 MCP 降级

**目标**：减少重复调用，并在实时服务失败时提供可追溯降级。

**范围**：

- 根据 Stage 1 实测决定缓存介质、TTL、容量、清理和持久化路径，并补充 ADR；
- 缓存键包含工具、标准化参数、数据版本和策略；
- 区分 `live / cached / stale`；仅在实时失败时允许 stale；
- 缓存故障不得破坏实时主链路。

**退出条件**：有效命中、过期重取、实时失败转 stale、无 stale 失败四类同步串行测试通过。

### Stage 18：天气驱动规划

**目标**：天气从展示信息升级为规划输入。

**范围**：

- 仅将数据源预报范围内的天气输入 ItineraryPlannerAgent；
- 提供室内外候选标签或明确决策上下文；
- 恶劣天气优先调整室内外顺序，并在草案理由中说明；
- 无预报时只标记 unavailable，不用气候常识冒充预报。

**退出条件**：有预报、超预报范围、天气查询失败三类用例均符合真实性边界。

### Stage 19：地图真实路线与时间轴联动

**目标**：地图只绘制 RouteService 路径，并与时间轴形成双向对应。

**范围**：

- 封装 `TripMap.vue`；
- 分天路线颜色、酒店起终点和天序号标记；
- 删除点位直线连接；必要 polyline 缺失时显示不可用状态；
- 时间轴 hover/click 高亮地图段，地图段 hover/click 定位时间轴；
- 使用稳定的 `day_id + segment_id` 建立关联。

**退出条件**：地图上的每段路线均可映射回 RouteSegment；02 文档 6.3 的可联动、可追溯要求通过。

### Stage 20：地图 Key 风险治理

**目标**：降低 Web JS Key 公开后被滥用的风险。

**范围**：

- 在高德控制台配置精确来源白名单、合理配额和最小权限；
- 区分前端 Web JS Key 与后端服务 Key，禁止混用；
- 更新 `.env.example` 和部署检查清单，日志/文档不出现真实 Key；
- 增加启动时缺失配置检查。

**退出条件**：非白名单来源不可使用前端 Key；后端 Key 不进入前端构建产物。控制台配置需留人工验收记录。

### Stage 21：前端代理统一

**目标**：在新增 SSE、PDF 和批量图片 API 前统一所有前端业务请求入口。

**范围**：统一 trip、map、poi、image、stream 和 report 请求配置；区分开发代理和生产 base URL；删除绕过 Vite 代理的硬编码地址，并为 JSON、流式响应和 multipart 上传保留独立请求配置。

**退出条件**：代码扫描无业务 API 硬编码 host；开发与生产构建配置均通过冒烟测试。

### Stage 22：多天独立折叠与全部展开

**目标**：各天天级卡片互不干扰，并确保任意 1～3 天可以同时完整展开。

**范围**：默认全部折叠，支持单独展开、全部展开和全部折叠，以稳定 day ID 保存展开状态；不改页面整体布局。

**退出条件**：1、2、3 天行程均可全部展开；展开/折叠任一天不影响其他天；当前 TripPlan 更新时无旧状态串扰。

### Stage 23：图片批量接口

**目标**：将 N 个前端图片 HTTP 请求收敛为单次批量请求。

**范围**：

- 后端接受去重后的点位 ID/关键词列表并同步处理，批量返回映射；
- 设置单批上限、超时和单项错误，不因一张失败导致整批失败；
- 服务端复用缓存；前端按一次行程一次或分块请求。

**退出条件**：典型报告的浏览器图片 API 请求数显著收敛；部分失败仍能渲染其他图片。

### Stage 24：图片加载优化

**目标**：保证天级卡片全部展开及导出准备时，图片加载状态准确可判断。

**范围**：图片懒加载、固定尺寸骨架屏、轻量占位图、有限次数重试和最终 fallback；提供导出流程可等待的 `imagesReady` 状态，重试耗尽也必须以明确 fallback 结束等待。

**退出条件**：慢网、404、超时和重试耗尽场景可预测；全部展开后图片均进入成功或最终 fallback 状态，不存在导出流程无限等待。

### Stage 25：导出重构与 PDF 截图就绪验收（独立难点 Stage）

**目标**：先证明完整行程页可以准确生成单一 PDF Blob，再开发浏览器下载和后端保存。

**范围**：

- 抽取共享的导出页面准备、截图、分页、文件命名和错误恢复逻辑，消除图片/PDF 重复代码；
- 在独立导出容器中展开全部 1～3 天天级卡片，并等待字体、图片成功/fallback 和地图渲染完成；
- 地图必须包含酒店、点位和真实路线信息；地图截图失败必须显式报错或使用明确占位，不得静默导出空白地图；
- 实现卡片感知的 A4 跨页处理，避免标题、时间轴事件、点位卡片和图片被页边界任意截断；
- 生成且只生成一次 `Blob`，通过自动化/人工基准验证内容；本 Stage 不接后端保存，也不正式启用“下载并加入历史”流程。

**退出条件**：1、2、3 天报告在全部展开下均包含完整图片、地图、时间轴和餐饮信息；跨页抽查无关键卡片截断、重复或遗漏；导出后页面状态可恢复。未通过时阻断 Stage 26、27。

### Stage 26：固定磁盘目录 PDF 存储与读取

**目标**：在 PDF Blob 已验证准确后，接收同一 PDF 并保存到固定磁盘目录。

**范围**：

- 后端提供 PDF 上传、历史列表和下载接口；
- PDF 写入配置项指定的固定磁盘目录，开发环境默认建议为项目下的 `data/trip_reports/`，该目录不提交 Git；
- 历史列表请求每次扫描固定目录中的全部 `.pdf` 文件，并根据安全文件名和文件修改时间生成展示信息，不依赖数据库；
- 文件名包含城市、行程日期、导出时间和防冲突短 ID；同名文件不得静默覆盖；
- 校验 PDF 文件头、MIME、大小上限和安全文件名，下载接口防止路径穿越。

**不做**：不保存或恢复结构化 TripPlan，不支持从历史 PDF 继续对话；维持当前本地/单用户目录方案，多用户隔离留待后续。

**退出条件**：服务重启后目录内全部合法 PDF 仍可列出和下载；上传、列表、下载、非法文件和路径穿越测试通过。

### Stage 27：Pinia、双向导航与 PDF 历史入口

**目标**：统一当前行程状态，并正式接入已验证 PDF Blob 的本地下载和固定目录保存。

**范围**：

- Pinia 管理当前 TripPlan、加载、对话状态和跨组件共享数据，不包含手动编辑状态；
- sessionStorage 只保存当前标签页的 TripPlan 临时副本；关闭标签页后不形成结构化历史；
- 行程页始终可返回首页；首页仅在存在有效当前 TripPlan 时显示“返回当前行程”，否则直接访问行程页返回首页；
- 用户点击“导出行程为 PDF”后调用 Stage 25 的单一 Blob：先触发浏览器下载，再上传完全相同的 Blob 到固定目录；
- 历史列表读取固定目录中的全部 PDF，仅展示名称、文件时间和下载入口；
- 本地下载成功但上传失败时提示“已下载到本地，但未加入历史”。

**退出条件**：初次进入、双向跳转、刷新恢复、关闭会话、导出/未导出、下载成功但上传失败、重新打开网页历史下载场景通过；历史 PDF 不能作为可编辑行程打开。

### Stage 28：SSE 同步阶段进度

**目标**：在保持后端处理同步串行的前提下，用真实阶段事件替换 `setInterval` 假进度。

**范围**：

- 事件顺序与真实调用链一致：数据检索 → 分级候选筛选 → 行程草案 → 路线计算/混合选择 → 时间轴与预算 → 校验/修复循环 → 完成或失败；
- 采用单请求流式响应或等价的串行任务协议，不启动并行查询或并发 Agent；
- 增加心跳、断连处理和最终结果；Coordinator 只发真实阶段状态，不伪造百分比；
- 前端展示阶段状态并移除假进度定时器。

**退出条件**：正常、修复循环、阶段失败和客户端断连四类集成测试通过；断连不产生重复正式结果。

### Stage 29：PC 显示稳定性与可读性

**目标**：在既定 PC 范围内完成局部视觉收敛。

**范围**：验证目标桌面宽度下时间轴、地图、长名称、缺失字段、partial/degraded 提示、全部天展开和 PDF 历史入口；只做局部 CSS 和信息可读性修正。

**不做**：不做移动端/平板适配，不整体重构页面布局。

**退出条件**：约定的 PC 浏览器与宽度矩阵人工验收通过，无横向溢出和关键状态遮挡。

### Stage 30：行前须知

**目标**：增加旅途级单次展示的预约、穿着和天气提示。

**范围**：

- ItineraryPlannerAgent 通过独立的受限结构化输出生成建议性须知并标记 `generated`；
- 天气提示只能引用可用预报，不将气候建议写成确定天气；
- 页面底部单次展示，不在每天重复；
- 预约事实无来源时使用条件式建议，不编造开放时间或预约政策。

**退出条件**：三类信息均有明确真实性标签；远期天气和缺失预约数据不会生成确定性事实。

### Stage 31A：无状态修订契约与完整性令牌（P2 后、P3 前）

**目标**：在不保存结构化行程数据库的前提下，建立可信的当前行程修订输入输出契约。

**范围**：

- 定义 `TripRevisionRequest`：当前 `TripPlan`、自然语言指令、当前 `revision_no` 和 `plan_token`；
- 生成与校验签名完整性令牌，令牌绑定 TripPlan 内容摘要和修订序号，防止客户端篡改事实字段；
- 后端不保存 TripPlan、对话消息或结构化版本历史；每次请求以客户端提交且验签通过的当前 TripPlan 为基准；
- 响应返回完整新版 TripPlan、新修订序号和新令牌；失败只返回错误，不修改客户端当前行程；
- 同一页面禁止并行提交修订，迟到响应因修订序号不匹配不得覆盖较新结果。

**不做**：不解析自然语言，不执行重规划，不保存服务端版本，不开发聊天 UI。

**退出条件**：合法令牌、篡改 TripPlan、过期修订序号、重复提交和失败保留当前行程测试通过。

### Stage 31B：对话指令结构化解析（独立难点 Stage）

**目标**：将自然语言调整请求约束为有限、可验证的 revision intent。

**范围**：

- 第一版只支持明确的调整类型：增删/替换点位、调整日期或顺序、更换酒店/餐饮、修改交通偏好；
- 定义各 intent 的 Pydantic 判别联合和字段校验；
- Agent 只解析意图和用户约束，不直接生成修改后的 TripPlan；
- 未知、越界或歧义指令返回结构化澄清状态，不猜测执行；
- 限制单次请求的操作数量，复合请求超限时要求拆分。

**退出条件**：每类白名单指令、歧义指令、越界指令和 Prompt 非法输出均有测试；解析阶段不修改当前 TripPlan。

### Stage 31C：修订重算与校验闭环（独立难点 Stage）

**目标**：执行结构化修订，并保证所有受影响事实和计算结果重新生成、重新校验。

**范围**：

- 根据 intent 计算最小受影响范围，但不复用已经失效的路线、时间或预算；
- 复用 TripCoordinator，按需重新检索候选、规划路线、计算时间/预算并校验；
- 成功后返回完整新版 TripPlan 和新完整性令牌，由前端原子替换当前状态；失败时返回原因且不产生部分更新；
- 点位、酒店、路线、时间或预算变动不得存在前端直改或跳过校验的旁路。

**不做**：不支持任意开放式工作流、多方案自动对比或未经白名单定义的修改。

**退出条件**：各类修订均触发正确的失效与重算范围；无法校验的结果不会返回为新版当前行程。

### Stage 32：对话式迭代前端与当前会话恢复

**目标**：为 Stage 31A～31C 提供可用的持续调整入口。

**范围**：

- 报告页增加局部对话入口，不重构整个页面布局；
- 展示用户指令、处理中状态、澄清、失败和版本更新；
- 新版本返回后原子替换 Pinia 当前行程，并同步时间轴与地图；
- 刷新后从 sessionStorage 恢复当前 TripPlan、修订序号和完整性令牌；对话消息和草稿只保存在当前会话；
- 关闭会话后当前结构化行程和对话可丢弃，已保存的历史 PDF 不用于恢复对话；
- 防重复提交，并处理修订序号冲突和令牌失效提示。

**退出条件**：连续两次调整、刷新恢复、调整失败、修订冲突、关闭会话后无结构化历史五类端到端用例通过；前端不直接修改当前 TripPlan 中的事实或计算字段。

### Stage 33：统一配置管理

**目标**：最后收敛 LLM、高德、Unsplash、缓存、PDF 报告目录和前端 API 配置。

**范围**：

- 后端统一经 `Settings` 读取、校验和脱敏展示配置；
- 前端仅保留可公开的 `VITE_*` 配置；
- 提供完整 `.env.example`、启动校验、环境分层说明；
- 删除重复读取和不再使用的环境变量；
- 明确 Key 轮换与部署检查步骤。

**退出条件**：缺失必需配置时启动快速失败且提示明确；代码扫描无散落配置读取和真实密钥。

## 五、关键质量门禁

### 5.1 Gate A：真实数据链可用（Stage 4 后）

- POI、地理编码、天气、路线均能解析为结构化对象；
- 外部格式异常不会静默返回成功；
- fixture 契约测试覆盖真实响应和主要异常形态。

未通过时，不进入 Agent 新链路开发。

### 5.2 Gate B：核心规划闭环可用（Stage 13 后）

- Agent 仅引用候选 ID；
- 规划顺序符合一级主要点位、二级酒店、三级便餐，依附关系可追溯；
- 路线、时间、预算均由确定性服务提供；
- 1～3 日计划满足早餐、用餐、缓冲、返店和末日边界；删点后新相邻路线已重新计算；
- `complete / partial / failed` 与 `degraded` 语义正确；
- 新链路 shadow 结果通过，旧 API 兼容适配仍可工作；此 Gate 不单独触发生产切换。

未通过时，不进入大规模前端功能开发。

### 5.3 Gate C：可靠性可上线（Stage 20 后）

- 无虚构 fallback；
- 缓存与 stale 降级可追溯；
- 错误分级、天气范围和地图路线真实性通过测试；
- 前后端 Key 隔离，控制台限制已完成。
- 新 TripCoordinator 链路已在 Stage 16 切为默认，生产路径不存在虚构 fallback 和手动编辑旁路。

### 5.4 Gate D：P2 体验闭环（Stage 30 后）

- 用户导出的 PDF 在完成浏览器下载后还会保存到固定目录，重新打开网页时可列出并下载；未导出行程不进入历史；SSE 反映真实状态；
- 图片、导出、代理、折叠和 PC 显示回归通过；
- 1～3 天天级卡片可全部展开，PDF 中图片、地图和跨页内容完整；所有外部查询保持同步串行。

只有 Gate D 通过后才启动 P0-9。

### 5.5 Gate E：对话迭代可上线（Stage 32 后）

- 所有实质调整走后端重算和校验；
- 修订失败不覆盖当前行程，刷新时当前会话恢复可靠；
- 前端没有仅修改本地正式数据的旁路。

## 六、测试策略

### 6.1 后端测试层级

- **单元测试**：解析器、去重、状态组合、TimelineCalculator、BudgetCalculator、Validator、缓存键与 TTL；
- **契约测试**：使用脱敏 MCP fixture 检测返回格式漂移；
- **Agent 边界测试**：未知 ID、非法字段、非 JSON、漏日期、重复点位和有限重试；
- **集成测试**：Coordinator 从候选到 TripPlan，外部服务全部使用可控 fake；
- **API 测试**：错误码、partial/degraded、SSE、PDF 归档下载、令牌校验和 revision 冲突；
- **少量在线冒烟**：只在显式配置测试 Key 时运行，不作为普通 CI 的稳定性依赖。

### 6.2 前端测试层级

- 类型检查与生产构建；
- Pinia store、API 适配和导出工具单元测试；
- 时间轴、状态提示、折叠、图片失败和聊天状态组件测试；
- 核心端到端流程：生成 → 首页/行程页双向跳转 → 导出 PDF（本地下载 + 固定目录保存）→ 重新打开网页 → 历史下载 → 当前行程对话调整。

### 6.3 必测业务边界

1. 1 日游同时适用末日 18:00 规则；
2. 非末日酒店早餐 09:00～21:00；
3. 非末日美食级早餐 08:00～21:00；
4. 末日两种早餐均不晚于 18:00；
5. 每段路线外存在至少 30 分钟独立缓冲；
6. 每顿在外用餐至少 40 分钟；
7. 多偏好全部检索，美食偏好开/关行为不同；
8. 超过 3 天、超过 3 个偏好、日期天数不一致和非法枚举均被拒绝；
9. POI 无结果、路线缺失、天气超范围、LLM 返回未知 ID/额外字段；
10. 混合交通只能选择真实 route ID，删点后重新计算新相邻路线；
11. `complete + degraded`、`partial + degraded`、`failed` 的前后端一致展示；
12. 同一当前行程的重复/迟到修订响应不能静默覆盖较新结果。

## 七、发布与回滚策略

1. **按 Stage 合并**：每个 Stage 单独提交/合并，不将多个高风险 Stage 打成一次大发布。
2. **兼容层切换**：Stage 5～15 期间新链路只做 shadow/功能开关验证；Stage 16 完成错误、状态和 fallback 治理后再设为默认。
3. **能力开关**：SSE、历史和对话迭代分别使用独立功能开关，便于小流量验证和快速关闭。
4. **报告文件兼容**：PDF 元数据格式只做向前兼容调整；回滚应用时不得删除或覆盖已有报告文件。
5. **外部服务保护**：外部查询保持同步串行，并配置超时、限流和调用统计；发现配额异常时收紧调用预算，不放宽真实性和失败规则。
6. **旧代码清理时机**：旧 `MultiAgentTripPlanner`、旧 schema、假进度和 sessionStorage 唯一路径，分别在对应替代能力稳定后单独清理，不夹带在功能 Stage 中。

## 九、初版待评审决策

以下事项不阻断本初版，但应在对应 Stage 开始前定稿：

1. Stage 1 实测后，确认 MCP 串行稳定性、路线工具选择和高德调用预算；
2. Stage 8 前，验证目标模型对 JSON mode/`response_format` 的实际兼容性；不改变“字符串适配 + `extra="forbid"` 校验 + 有限重试”的既定方案；
3. Stage 17 前，根据部署形态确定缓存介质、TTL、容量和清理策略；
4. Stage 26 前，确认 PDF 固定磁盘目录、容量上限、清理和备份策略；本计划不引入 SQLite，并维持本地/单用户范围；
5. Stage 28 前，确认同步串行 SSE 的单请求流式协议和断连行为，不引入后台并行任务；
6. Stage 31B 前，评审首版 revision intent 白名单，避免以“自然语言无限能力”扩大范围。

上述决策均应形成简短 ADR，并回填本计划；不得把尚未验证的假设直接固化为生产实现。
